## エグゼクティブサマリー

本白書は、**NIST耐量子暗号（PQC）署名検証のゼロ知識STARK証明**に関する、世界初の包括的な実装と比較分析を提示します。NISTがFIPS 204（Dilithium）、FIPS 203（Kyber）、FIPS 205（SPHINCS+）を標準化したことを受け、暗号コミュニティは量子耐性アルゴリズムへの移行を進めています。この環境において、ゼロ知識証明システム内でPQC演算を検証する能力は、プライバシー保護アプリケーションにとって不可欠となります。

### 主要成果

| 指標 | SP1（汎用zkVM） | Plonky2（カスタムSTARK） |
|------|-----------------|-------------------------|
| **スケーリング** | O(n^0.96) | O(n^0.77) |
| **証明生成時間（N=256）** | 7ms実行 | 30.2ms |
| **コスト/証明** | $0.0009 | N/A（自己ホスト） |
| **証明サイズ** | ~200KB | 106KB |
| **開発工数** | 低（Rustコード） | 高（回路設計） |

**主要知見:** SP1とPlonky2の両アプローチで、Dilithium NTT演算のZK検証に成功し、プライバシー保護型耐量子署名検証の実現可能性を実証しました。SP1とPlonky2の選択は、具体的なユースケース要件に依存します：**SP1は開発速度とコスト効率に優れ**、**Plonky2は特化アプリケーション向けに優れた証明生成速度とコンパクトな証明サイズ**を提供します。

## 1. 序論

### 1.1 量子脅威

大規模量子コンピュータの出現は、従来の暗号システムに対する存在論的脅威をもたらします。Shorのアルゴリズムは大きな整数の因数分解と離散対数の計算を効率的に行うことができ、RSAと楕円曲線暗号を時代遅れにします。NISTの耐量子暗号標準化プロジェクトは、3つの最終標準を確定しました：

- **FIPS 204（ML-DSA/Dilithium）:** 格子ベースのデジタル署名方式
- **FIPS 203（ML-KEM/Kyber）:** 格子ベースの鍵カプセル化メカニズム
- **FIPS 205（SLH-DSA/SPHINCS+）:** ハッシュベースの署名方式

### 1.2 ゼロ知識と耐量子暗号の融合

ゼロ知識証明は、ある値の知識を、その値自体を明かすことなく証明することを可能にします。PQC検証と組み合わせることで、以下の強力なプリミティブが生まれます：

- **秘匿クレデンシャル検証:** 署名や公開鍵を明かさずに、有効なDilithium署名の所有を証明
- **秘匿トランザクション:** 暗号化された計算内でPQC署名による認可を検証
- **クロスチェーンブリッジ:** 外部チェーンからのPQC署名をトラストレスに検証
- **規制遵守:** 機密データを公開せずに暗号学的コンプライアンスを証明

### 1.3 プロジェクト範囲

本プロジェクトは、Dilithium演算のZK検証に対する2つの異なるアプローチを実装・ベンチマークします：

1. **SP1（Succinct zkVM）:** 汎用RISC-Vゼロ知識仮想マシン
2. **Plonky2（カスタムSTARK）:** Goldilocksフィールドを使用した手動最適化算術回路

## 2. 実装詳細

### 2.1 `zk-dilithium-ntt` クレート

コアライブラリは、Dilithiumの数学的演算のSTARKフレンドリーな表現を実装します：

```
src/

├── constants/     # Dilithiumパラメータ（Q, R, ZETA等）
├── trace/         # 実行トレース生成
├── prover/        # STARK証明構築

├── sphincs/       # FIPS 205サポート（Phase IV-C）
└── formal_verification/  # 健全性証明
```

#### 主要暗号定数

| 定数 | 値 | 説明 |
|------|-----|------|
| Q | 8,380,417 | Dilithium素数モジュラス（2²³ - 2¹³ + 1） |
| R | 2³² | Montgomery定数 |
| N | 256 | NTT多項式次数 |
| ZETA | 1753 | Q上の原始512乗根 |
| NEG_Q_INV_MOD_R | 4,236,238,847 | Montgomery約減定数 |

#### 実装済みゲート

| ゲート | 演算 | 制約次数 |
|--------|------|----------|
| **NTTバタフライ** | (a', b') = (a + ωb, a - ωb) | 2 |
| **FMA** | c = a × b + d（Montgomery） | 3 |
| **Truncation** | 上位/下位ビット分解 | 2 |

| **PRC** | 16ビットチャンク順列範囲チェック | 2 |

### 2.2 SP1実装アプローチ

SP1は標準的なRustコードをRISC-V zkVM内で実行し、正しい実行のSTARK証明を自動生成します。

**アーキテクチャ:**
```
sp1-bench/
├── program/       # ゲストプログラム（zkVM内で実行）
│   └── src/main.rs
└── script/        # ホストスクリプト（証明生成のオーケストレーション）
    └── src/main.rs
```

**主要設計決定:**

1. **直接Montgomery算術:** ネイティブu64演算を使用した`montgomery_multiply`の実装
2. **事前計算済みトゥイドル因子:** NTTバタフライ用のコンパイル時定数配列
3. **バッチ処理:** 償却のための複数演算を1証明に含める
4. **サイクル計上:** zkVM命令カウントの精密測定

**ゲストプログラムのコアループ:**
```rust
// SP1でのMontgomery乗算
fn montgomery_multiply(a: u64, b: u64) -> u64 {
    let t = (a as u128) * (b as u128);
    let m = ((t as u64).wrapping_mul(NEG_Q_INV_MOD_R)) as u128;
    let result = (t + m * (Q as u128)) >> 32;
    if result >= Q as u128 { (result - Q as u128) as u64 } else { result as u64 }
}
```

### 2.3 Plonky2実装アプローチ

Plonky2はGoldilocksフィールド（p = 2⁶⁴ - 2³² + 1）と手動設計の算術回路を使用します。

**アーキテクチャ:**
```
plonky2-bench/
├── Cargo.toml           # 依存関係（plonky2 v1.1）
├── rust-toolchain.toml  # Nightly Rust要件
└── src/main.rs          # 回路定義とベンチマーク
```

**主要設計決定:**

1. **Goldilocksフィールド埋め込み:** p >> Qのため、Dilithium算術が自然に埋め込み可能
2. **DilithiumArithmeticガジェット:** 最適化された乗算/FMA演算
3. **バッチ検証回路:** 単一証明内に複数NTT演算
4. **Cooley-Tukey構造:** 完全な基数2 NTTバタフライネットワーク

**回路ビルダーの例:**
```rust
fn build_butterfly(
    builder: &mut CircuitBuilder<F, D>,
    a: Target, b: Target, twiddle_idx: usize
) -> (Target, Target) {
    let twiddle = builder.constant(F::from_canonical_u64(TWIDDLE_FACTORS[twiddle_idx]));
    let wb = builder.mul(twiddle, b);
    let a_prime = builder.add(a, wb);
    let b_prime = builder.sub(a, wb);
    (a_prime, b_prime)
}
```

## 4. アーキテクチャ推奨事項

### 4.1 決定マトリクス

| 評価基準 | SP1推奨 | Plonky2推奨 |
|----------|---------|-------------|
| **開発時間** | ✓ Rustを直接記述 | ✗ 回路専門知識が必要 |
| **イテレーション速度** | ✓ 標準デバッグ | ✗ Witness/制約デバッグ |
| **証明サイズ** | ✗ ~200KB | ✓ ~100KB |
| **証明速度（小N）** | ✗ VMオーバーヘッド | ✓ 直接回路 |
| **証明速度（大N）** | ≈ 同等 | ≈ 同等 |
| **コスト（クラウド証明）** | ✓ $0.0009/証明 | ✗ 自己ホストのみ |
| **再帰** | ✓ ネイティブサポート | ✓ ネイティブサポート |
| **監査範囲** | ✓ 標準Rust | ✗ カスタム回路 |

### 4.2 ユースケース別推奨

#### SP1を選択すべき場合:

1. **ラピッドプロトタイピング:** PQC-ZK統合を迅速に検証する必要がある
2. **複雑なロジック:** 条件分岐、ループ、動的データを含む検証
3. **コスト重視:** Succinct Networkの従量課金モデルを活用
4. **チーム構成:** ZK回路の専門知識がない開発者
5. **保守性優先:** 生の性能よりも長期的な保守性

**例:** プライバシー保護型アイデンティティシステムを構築するスタートアップ。Dilithium署名の検証が必要で、開発速度とSuccinct Network統合が優先事項。

#### Plonky2を選択すべき場合:

1. **性能が最重要:** 100ms未満の証明生成が必要
2. **証明サイズ制約:** 帯域制限環境（IoT、モバイル）
3. **大量処理:** スケールでの自己ホスト証明（1日数百万証明）
4. **特化演算:** 固定的で明確に定義された検証ロジック
5. **研究コンテキスト:** 新規回路最適化の探求

**例:** PQC署名によるトランザクションバッチを検証するLayer 2ロールアップ。証明サイズがオンチェーンコストに直接影響し、ボリュームがカスタム回路開発を正当化。

## 5. セキュリティ考慮事項

### 5.1 健全性保証

SP1とPlonky2の両方が、標準的な暗号学的仮定の下で計算的健全性を提供します：

| 特性 | SP1 | Plonky2 |
|------|-----|---------|
| **健全性誤り** | 2^-100 | 2^-100 |
| **ハッシュ関数** | Poseidon | Poseidon |
| **フィールドサイズ** | BabyBear（31ビット） | Goldilocks（64ビット） |
| **FRIセキュリティ** | 100ビット | 100ビット |

### 5.2 形式検証

`formal_verification`モジュールは機械検証可能な仕様を提供します：

```rust
pub struct MontgomeryFMASpec {
    pub input_a: u64,
    pub input_b: u64,
    pub input_c: u64,
    pub expected_output: u64,
    // 証明: output ≡ (a × b × R⁻¹ + c) mod Q
}
```

以下の健全性証明が利用可能です：
- Montgomery乗算の正しさ
- NTTバタフライの可逆性
- FMAアキュムレータ境界
- ノルムチェック完全性

### 5.3 サイドチャネル考慮事項

STARK証明は本質的にいくつかのサイドチャネル耐性を提供します：
- **タイミング:** 証明生成時間はデータ非依存（固定回路）
- **電力:** ソフトウェア証明者には適用外
- **キャッシュ:** Witness生成は情報を漏洩する可能性あり；定数時間実装を使用

## 8. ECDSAハイブリッド分析

### 8.1 ハイブリッド検証の課題

耐量子暗号への移行期間中、多くのシステムは**従来型（ECDSA）と耐量子（Dilithium）署名の両方を同時に検証する**必要があります。このハイブリッドアプローチは多層防御を提供します：一方のアルゴリズムが破られても、もう一方がセキュリティを提供します。

### 8.2 SP1 ECDSA性能（k256クレート）

実際のECDSA検証を、SP1のzkVM内でk256クレートを使用してベンチマークしました：

| 演算 | サイクル | Dilithiumのみとの比較 |
|------|----------|----------------------|
| Dilithium NTT（N=256） | 60.56K | ベースライン |
| ECDSA secp256k1検証 | 4.62M | **76.3倍** |
| ハイブリッド（両方） | ~4.68M | **77.3倍** |

**主要知見:** ECDSA検証がハイブリッド証明コストを支配し、**署名あたり4.62Mサイクル**を要します。これはDilithiumのみの検証と比較して1343%のオーバーヘッドを表します。

### 8.3 Plonky2 EC-Gadget実装

SP1のECDSAオーバーヘッドに対処するため、Plonky2でカスタムsecp256k1 EC-Gadgetを実装しました：

**アーキテクチャ:**
```
secp256k1_gadget.rs
├── NonNativeTarget     # 256ビット整数を4×64ビットリムで表現
├── NonNativeArithmetic # Goldilocksフィールド上の加算/減算/乗算
├── ECPointTarget       # 射影座標（X:Y:Z）
├── Secp256k1Gadget    # 点演算（倍算、加算、スカラー乗算）
└── EcdsaVerificationCircuit # 完全なECDSA検証構造
```

**性能比較（64回のフィールド乗算）:**

| 指標 | SP1汎用 | Plonky2 EC-Gadget |

## 9. 量子安全STARK

### 9.1 セキュリティパラメータ分析

標準のPlonky2設定は**約50ビットの量子セキュリティ**を提供しますが、これは長期的なセキュリティには不十分です。強化されたFRIパラメータを持つ量子耐性設定を実装しました。

### 9.2 FRIクエリ数最適化

STARKのコアセキュリティパラメータはFRIクエリ数です。各クエリは`rate_bits`の古典セキュリティを提供します：

| セキュリティレベル | 目標量子セキュリティ | FRIクエリ | Rate Bits | PoW Bits | 合計古典 |
|-------------------|---------------------|-----------|-----------|----------|----------|
| Standard | 50ビット | 28 | 3 | 16 | 100ビット |
| **High** | **100ビット** | **76** | **3** | **20** | **248ビット** |
| Paranoid | 128ビット | 95 | 4 | 24 | 404ビット |

**計算式:** `量子セキュリティ ≈ (クエリ数 × rate_bits + pow_bits) / 2`

### 9.3 Poseidonハッシュ量子セキュリティ

Plonky2はGoldilocksフィールドでPoseidonハッシュを使用します：

| 特性 | 設定 | 古典 | 量子 |
|------|------|------|------|
| 状態幅 | 12要素（768ビット） | - | - |
| キャパシティ | 4要素（256ビット） | - | - |
| **衝突耐性** | 誕生日境界 | 128ビット | **64ビット** |
| **原像耐性** | フルキャパシティ | 256ビット | **128ビット** |

**重要な洞察:** STARK証明では**原像耐性が主要**です。Poseidonの128ビット量子原像耐性は、100ビット量子セキュリティ目標に十分です。

### 9.4 性能ペナルティ測定

標準設定と量子耐性設定を比較する実ベンチマーク：

| 指標 | 標準設定 | 量子100ビット設定 | ペナルティ |
|------|----------|-------------------|-----------|
| ビルド時間 | 9.2ms | 5,427.8ms | **590倍** |
| 証明生成時間 | 12.7ms | 10,719.4ms | **844倍** |
| 検証時間 | 1.7ms | 8.6ms | **5.1倍** |
| 証明サイズ | 97KB | 410KB | **4.2倍** |
| 量子セキュリティ | ~50ビット | ~124ビット | **+74ビット** |

### 9.5 性能ペナルティの正当性

590-844倍の証明生成オーバーヘッドは以下の場合に正当化されます：

1. **長期セキュリティ:** 今日生成された証明が、量子コンピュータが存在する数十年後に検証される可能性
2. **高価値トランザクション:** 最大限の保証を必要とする金融、法的、規制関連の証明
3. **重要インフラ:** 政府、医療、防衛アプリケーション
4. **コンプライアンス要件:** NIST PQCコンプライアンスを義務付ける組織

**推奨:** アーカイブ証明には量子耐性設定を使用；時間限定の有効性を持つ一時的証明には標準設定を使用。
