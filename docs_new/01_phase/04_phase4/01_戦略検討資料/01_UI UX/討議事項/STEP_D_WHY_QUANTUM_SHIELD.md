# 第1章：脅威 - なぜ今、耐量子暗号が必要か

## 1.1 量子コンピュータとは何か

量子コンピュータは、従来のコンピュータとは根本的に異なる原理で動作します。

従来のコンピュータが「0か1か」という2値で計算するのに対し、量子コンピュータは「0でもあり1でもある」という**重ね合わせ状態**を利用します。これにより、特定の問題を従来の何兆倍もの速度で解くことができます。

**ユーザーに伝えるべきこと：**
> 量子コンピュータは「すべてを高速化する」わけではありません。特定の問題—特に暗号を解く問題—で圧倒的な能力を発揮します。

## 1.2 今の暗号が危険な理由

現在のブロックチェーン（Bitcoin、Ethereum等）は、主に以下の暗号方式に依存しています：

| 暗号方式 | 用途 | 量子コンピュータへの耐性 |
|---------|------|:----------------------:|
| ECDSA（楕円曲線署名） | トランザクション署名 | ❌ **脆弱** |
| RSA | 一部の認証 | ❌ **脆弱** |
| SHA-256 | ハッシュ計算 | ⚠️ 弱体化 |

### なぜ危険なのか

量子コンピュータは**Shor（ショア）のアルゴリズム**を使うことで、楕円曲線暗号（ECDSA）やRSAを**数時間で破ることができます**。

これは理論上の話ではありません。量子コンピュータの実用化は「もし」の問題ではなく「いつ」の問題です。

```
現在の予測:
├── 2028-2030年: 暗号学的に意味のある量子コンピュータの登場
├── 2030-2035年: 商用量子コンピュータの普及
└── ???: 既存の暗号が一夜にして無効化される「Q-Day」
```

## 1.3 「Harvest Now, Decrypt Later」攻撃

最も深刻な脅威は、**今すでに始まっています**。

悪意のある攻撃者は、現在暗号化されたデータを大量に収集（Harvest）し、将来量子コンピュータが実用化された時に一気に復号（Decrypt）することができます。

```
2024年（現在）              2030年（将来）
    │                           │
    │ 暗号化されたデータを収集    │ 量子コンピュータで復号
    │    ↓                      │    ↓
    │ [暗号化されたトランザクション]  → [秘密鍵の抽出]
    │ [署名データ]                   → [資産の窃取]
    │ [通信記録]                     → [完全な解読]
```

**ユーザーに伝えるべきこと：**
> あなたの資産を守る「鍵」は、今この瞬間も攻撃者に記録されている可能性があります。量子コンピュータが実用化された瞬間、それらの記録から鍵を復元され、資産を奪われるリスクがあります。

## 1.4 なぜブリッジが特に危険か

クロスチェーンブリッジは、異なるブロックチェーン間で資産を移動させる仕組みです。そしてブリッジは、暗号通貨業界で**最も狙われる標的**です。

| 年 | 事件 | 被害額 |
|----|------|--------|
| 2022 | Ronin Bridge | $620M |
| 2022 | Wormhole | $320M |
| 2022 | Nomad | $190M |
| 2023 | Multichain | $130M |

**なぜブリッジが狙われるのか：**

1. **資産が集中している** - ブリッジには大量の資産がロックされている
2. **複雑性が高い** - 複数のチェーンをまたぐため、攻撃面が広い
3. **秘密鍵への依存** - 多くのブリッジは少数の秘密鍵に依存している

量子コンピュータが実用化されると、これらの秘密鍵は**即座に**危険にさらされます。

# 第2章：解決策 - Quantum Shieldのアプローチ

## 2.1 NIST準拠の耐量子暗号

Quantum Shieldは、米国国立標準技術研究所（NIST）が策定した**耐量子暗号標準**に完全準拠しています。

| 用途 | 採用アルゴリズム | NIST標準 | 量子耐性 |
|------|----------------|---------|:-------:|
| ユーザー署名 | **Dilithium-III** | FIPS 204 | ✅ |
| Prover署名 | **SPHINCS+-128s** | FIPS 205 | ✅ |
| ハッシュ | **SHA3-256** | FIPS 202 | ✅ |

**なぜこれらの暗号が安全なのか：**

- **Dilithium**: 格子問題（Lattice Problem）に基づく。量子コンピュータでも解読に数千年かかる
- **SPHINCS+**: ハッシュのみに基づく。量子アルゴリズムでも指数関数的な時間が必要
- **SHA3-256**: Grover攻撃に対しても128ビットの安全性を維持

**ユーザーに伝えるべきこと：**
> Quantum Shieldは、世界最高峰の標準化機関NISTが「量子コンピュータ時代でも安全」と認定した暗号のみを使用しています。

## 2.2 なぜ従来の暗号を完全に排除するのか

多くのシステムは「後方互換性」を理由に、従来の暗号（ECDSA等）を残しています。

**Quantum Shieldは違います。**

私たちは**量子脆弱な暗号を一切使用しません**。

```
❌ 禁止アルゴリズム（Quantum Shieldでは使用不可）:
├── ECDSA (楕円曲線署名)
├── RSA
├── secp256k1 (Ethereumの標準曲線)
├── SHA-256 / SHA-2ファミリー
└── keccak256 (Solidityの標準ハッシュ)
```

**なぜ排除するのか：**

「1つでも脆弱な部分があれば、全体が脆弱になる」—これがセキュリティの原則です。

たとえDilithiumで署名しても、システムの一部にECDSAが残っていれば、攻撃者はその弱点を突くことができます。

**ユーザーに伝えるべきこと：**
> 「部分的な安全」は「安全ではない」と同じです。Quantum Shieldは、妥協のない完全な量子耐性を提供します。

## 2.3 Self-Custody：あなたの鍵は、あなただけのもの

Quantum Shieldの根本原則の一つが**Self-Custody（自己管理）**です。

```
┌─────────────────────────────────────────────────────────────┐
│                      Self-Custody の原則                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  あなたの秘密鍵は：                                          │
│  ✅ あなたのデバイスにのみ保存される                         │
│  ✅ 暗号化された状態で管理される                             │
│  ✅ ネットワークに送信されることは決してない                  │
│                                                             │
│  Quantum Shieldは：                                         │
│  ✅ 公開鍵のみを受け取る                                     │
│  ❌ 秘密鍵を預かることは決してない                           │
│  ❌ あなたに代わって署名することは決してない                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**なぜSelf-Custodyが重要か：**

従来のブリッジやカストディサービスでは、運営者が秘密鍵を管理しています。これは：

- 運営者のセキュリティに完全に依存する
- 運営者が悪意を持てば資産を持ち逃げできる
- 運営者がハッキングされれば全ユーザーの資産が危険に

**ユーザーに伝えるべきこと：**
> あなたの資産の「鍵」を他人に預けることは、家の鍵を見知らぬ人に預けるようなものです。Quantum Shieldでは、鍵はあなただけのものです。

# 第3章：安全装置 - なぜ24時間/7日待つのか

## 3.1 Time Lock：時間という盾

Quantum Shieldでは、資産を引き出す（Unlock）際に**待機時間**が必要です。

| シナリオ | 待機時間 | なぜこの時間か |
|---------|:-------:|--------------|
| 通常Unlock | **24時間** | 異常を検知し、対応するための最低限の時間 |
| 緊急Unlock | **7日間** | 十分な検証と、必要に応じたChallenge期間 |

**なぜ即座に引き出せないのか：**

「便利さ」と「安全性」はトレードオフの関係にあります。

```
即座に引き出せる場合：
├── 攻撃者が不正に資産を盗む
│   └── 誰も気づかない間に資産が消える
│       └── 取り返しがつかない
│
24時間の待機がある場合：
├── 攻撃者が不正に資産を盗もうとする
│   └── 24時間の間に異常を検知
│       └── Challengeで阻止
│           └── 攻撃者の資産（Stake）が没収される
```

**ユーザーに伝えるべきこと：**
> 24時間の待機時間は「不便さ」ではなく「安全装置」です。この時間があるからこそ、不正な引き出しを阻止できます。

## 3.2 Prover署名：第三者による検証

Unlockには、あなたの署名だけでなく、**Prover**と呼ばれる検証者の署名も必要です。

```
Unlock申請の流れ：

あなた                    システム                   Prover（5社）
  │                         │                          │
  │──── Unlock申請 ────────►│                          │
  │                         │──── 2社をランダム選出 ───►│
  │                         │                          │
  │                         │◄─── 検証 + 署名 ────────│
  │                         │                          │
  │                         │（2社の署名が揃ったら）    │
  │◄──── 24時間後に引出可能 │                          │
```

**なぜProverが必要か：**

- **単一障害点の排除**: あなたの秘密鍵が漏洩しても、Proverの署名がなければ引き出せない
- **分散された検証**: 5社のProverのうち2社がランダムに選ばれる。全員を買収するのは極めて困難
- **経済的インセンティブ**: Proverは$400K以上のStakeを預けており、不正行為は経済的な自殺行為

**ユーザーに伝えるべきこと：**
> あなたの資産を守るのは、あなた一人ではありません。独立した複数の検証者が、あなたの取引が正当であることを確認します。

## 3.3 Slashing：不正行為への経済的罰則

Proverが不正を働いた場合、彼らのStake（預け入れた資産）が**没収**されます。

```
┌─────────────────────────────────────────────────────────────┐
│                  Quadratic Slashing（二乗罰則）              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  不正Prover数    罰則率          例（$400K Stake）           │
│  ────────────   ──────────      ─────────────────           │
│     1社         10%             $40K 没収                   │
│     2社         40%  (2²×10%)   $160K 没収                  │
│     3社         90%  (3²×10%)   $360K 没収                  │
│     4社+        100%            全額没収                     │
│                                                             │
│  ※ 共謀するほど罰則が急激に増加する設計                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**なぜ二乗罰則か：**

1社だけの不正は「ミス」かもしれない。しかし複数社が同時に不正を働くのは「共謀」です。共謀は個別の不正より遥かに危険なため、罰則も急激に重くなります。

**ユーザーに伝えるべきこと：**
> 検証者（Prover）は、不正を働くと多額の資産を失います。彼らには「正直に行動する」強力な経済的動機があります。

## 3.4 Observer & Challenger：監視の目

誰でも**Observer**（監視者）や**Challenger**（異議申立者）になれます。

```
監視と異議申立のフロー：

異常なUnlock申請
       │
       ▼
┌──────────────┐
│  Observer    │  ← 誰でもなれる（Stakeなし）
│  24/7監視    │
└──────┬───────┘
       │ 異常検知！
       ▼
┌──────────────┐
│  Challenge   │  ← Bond（保証金）を預けて異議申立
│  異議申立    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  判定        │
├──────────────┤
│ 不正確定：   │ → 被害者保護 + Prover Slashing + 報告者報酬
│ 正当と判定： │ → Challenger のBond没収
└──────────────┘
```

**なぜこの仕組みがあるか：**

- **自己監視システム**: システムを監視するのは運営者だけではない
- **インセンティブ設計**: 不正を発見すると報酬がもらえる。だから監視する人がいる
- **悪意のChallenge抑止**: 虚偽のChallengeをするとBondを失う

**ユーザーに伝えるべきこと：**
> Quantum Shieldは、世界中の誰もが監視に参加できるオープンなシステムです。あなたの資産は、多くの目によって守られています。

# 第4章：2つのモデル - Enterprise vs Decentralized

## 4.1 なぜ2つのエディションがあるのか

Quantum Shieldには、2つの異なる提供形態があります。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Quantum Shield の2つの顔                        │
├────────────────────────────────┬────────────────────────────────────┤
│      Enterprise Edition        │       Decentralized Edition        │
├────────────────────────────────┼────────────────────────────────────┤
│                                │                                    │
│  対象: 金融機関、銀行、保険     │  対象: DeFi、ブリッジ、カストディ  │
│                                │                                    │
│  特徴:                         │  特徴:                             │
│  • 契約ベースの関係            │  • パーミッションレス参加          │
│  • 専用サポート、SLA           │  • コミュニティ主導                │
│  • 規制対応サポート            │  • トークンガバナンス              │
│  • 固定インフラ（安定性重視）  │  • 分散インフラ（検閲耐性重視）    │
│                                │                                    │
│  管理: QS財団（中央集権的）    │  管理: DAO/コミュニティ（分散型）  │
│                                │                                    │
└────────────────────────────────┴────────────────────────────────────┘
```

**なぜ分けるのか：**

顧客のニーズは一様ではありません。

- 銀行は「規制対応」と「責任の所在」を重視する
- DeFiプロジェクトは「検閲耐性」と「パーミッションレス」を重視する

同じ技術基盤の上に、異なるガバナンスと運用モデルを提供することで、両方のニーズに応えます。

## 4.2 共通の技術基盤

両エディションは、**同じセキュリティ保証**を持ちます。

| 項目 | Enterprise | Decentralized |
|------|:----------:|:-------------:|
| 耐量子暗号 | ✅ 同一 | ✅ 同一 |
| 24h/7d Time Lock | ✅ 同一 | ✅ 同一 |
| Quadratic Slashing | ✅ 同一 | ✅ 同一 |
| Self-Custody | ✅ 同一 | ✅ 同一 |
| オンチェーン透明性 | ✅ 同一 | ✅ 同一 |

**異なるのは「誰が管理するか」であり、「セキュリティレベル」ではありません。**

**ユーザーに伝えるべきこと：**
> どちらのエディションを選んでも、あなたの資産は同じレベルの量子耐性で保護されます。違いは、誰が運営に責任を持つかです。

## 4.3 段階的な分散化

Decentralized Editionは、一夜にして完全分散化されるわけではありません。

```
Phase 1-2: 準備段階
├── ノード: 4ノード固定（QS財団運営）
├── Prover: 招待制
└── ガバナンス: マルチシグ

Phase 3: 移行段階
├── ノード: 4ノード固定（一部外部参加）
├── Prover: Security Council承認制
└── ガバナンス: トークン投票開始

Phase 4+: 完全分散化
├── ノード: 4-21ノード（動的参加）
├── Prover: Stakeベース自動承認
└── ガバナンス: フルDAO
```

**なぜ段階的に行うのか：**

- **安全な移行**: 一度に全てを変えるとリスクが高い
- **学習と改善**: 各フェーズで問題を発見し、次に活かす
- **信頼の構築**: コミュニティが徐々に経験を積む

**ユーザーに伝えるべきこと：**
> 分散化は「目的地」ではなく「旅」です。安全に、確実に、一歩ずつ進んでいきます。

# 第5章：Core Principles（不変の原則）

## 5.1 5つの憲法

Quantum Shieldには、**いかなるガバナンス投票でも変更できない**原則があります。

```
┌─────────────────────────────────────────────────────────────┐
│              Core Principles（不変の原則）                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  CP-1: 完全量子耐性                                         │
│        NIST準拠の耐量子暗号のみを使用する                   │
│                                                             │
│  CP-2: Self-Custody                                         │
│        ユーザーの秘密鍵はユーザーのみが管理する             │
│                                                             │
│  CP-3: Time Lock存在                                        │
│        Time Lockを0にすることは不可                         │
│                                                             │
│  CP-4: Slashing存在                                         │
│        Slashingメカニズムの削除は不可                       │
│                                                             │
│  CP-5: 透明性                                               │
│        全てオンチェーンで検証可能                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**なぜ「不変」なのか：**

これらの原則を変更できてしまうと、システム全体のセキュリティが崩壊します。

例えば、Time Lockを0にできれば、不正検知の時間がなくなり、Challengeが機能しなくなります。

**ユーザーに伝えるべきこと：**
> これらの原則は、将来にわたってあなたの資産を守り続けることを保証します。誰も—私たちですら—これらを変更することはできません。

**ポイント：**

1. **プロセスの可視化**: 何が起きているかを明示
2. **理由の説明**: 「なぜ？」に答える手段を提供（❓アイコン）
3. **安心感の言葉**: 「安全に」「検証」など、セキュリティを感じる表現

## 6.3 不安を解消するコミュニケーション

ユーザーが感じる不安と、それを解消するUIコピー：

| ユーザーの不安 | 解消するコピー |
