## 1. 概要

### 1.1 Prover登録とは

Proverは、Unlock時にSPHINCS+署名を提供する署名サービス提供者。量子耐性のある署名をHSM内で生成し、ユーザーの資産引出しを承認する。

### 1.2 登録フローの種類

| モード | 対象 | 承認方式 | 所要時間 |
|--------|------|----------|----------|
| CONTRACT_BASED | Enterprise | 法的契約 + Admin承認 | 1-2週間 |
| FOUNDATION_INVITE | Decen Phase 1-2 | 財団招待 | 1-3日 |
| COUNCIL_VOTE | Decen Phase 3 | Council 3/9投票 | 7日 |
| STAKE_AUTO | Decen Phase 4+ | 自動承認 | 即時 |

### 1.3 現状の問題

🔴 **CDO指摘**: Prover登録のUI/UXが**完全欠如**
- UI_UX_FUNCTIONAL_REQUIREMENTS_JP.mdにProver登録画面フローがない
- HSM証明のアップロードUIがない
- 承認プロセスの可視性がない

#### HSMVerifier.sol（新規）

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

/// @title HSMVerifier
/// @notice HSM Attestation証明の検証
contract HSMVerifier {
    
    // サポートされるHSMベンダー
    mapping(bytes32 => bool) public supportedVendors;
    
    // FIPS 140-2 Level要件
    uint8 public constant MIN_FIPS_LEVEL = 3;
    
    struct AttestationData {
        bytes32 vendorId;
        uint8 fipsLevel;
        uint256 validUntil;
        bytes32 deviceId;
        bytes signature;
    }
    
    constructor() {
        supportedVendors[keccak256("THALES")] = true;
        supportedVendors[keccak256("ENTRUST")] = true;
        supportedVendors[keccak256("UTIMACO")] = true;
        supportedVendors[keccak256("AWS_CLOUDHSM")] = true;
        supportedVendors[keccak256("AZURE_DEDICATED_HSM")] = true;
    }
    
    function verify(bytes calldata attestation) 
        external view 
        returns (bool valid, string memory reason) 
    {
        AttestationData memory data = abi.decode(attestation, (AttestationData));
        
        if (!supportedVendors[data.vendorId]) {
            return (false, "Unsupported HSM vendor");
        }
        
        if (data.fipsLevel < MIN_FIPS_LEVEL) {
            return (false, "FIPS level too low");
        }
        
        if (block.timestamp > data.validUntil) {
            return (false, "Attestation expired");
        }
        
        return (true, "");
    }
}
```

## 8. エラーハンドリング

### 8.1 エラーコード

| Code | 説明 | 対応方法 |
|------|------|----------|
| `ERR_INVALID_HSM` | HSM検証失敗 | サポートベンダーのHSMを使用 |
| `ERR_INVALID_KEY` | SPHINCS+鍵不正 | 鍵形式を確認 |
| `ERR_INSUFFICIENT_STAKE` | Stake不足 | 最低Stake以上を送金 |
| `ERR_INVALID_MULTISIG` | マルチシグ不正 | 2-of-3を設定 |
| `ERR_MODE_MISMATCH` | モード不一致 | 現在のモードで登録 |
| `ERR_INVITE_EXPIRED` | 招待コード期限切れ | 新しい招待を取得 |
| `ERR_ALREADY_REGISTERED` | 既に登録済み | ダッシュボードへ |

## 9. 🔴 CDO/CIA指摘と解決策

### 9.1 CDO指摘

#### 指摘1: UI/UXの完全欠如

**問題**: 現在のUI_UX_FUNCTIONAL_REQUIREMENTS_JP.mdにProver登録画面フローがない

**解決策**: 本仕様書のセクション6で詳細UI/UXを定義
- 画面遷移図
- 各画面のワイヤーフレーム
- エラー状態のUI

#### 指摘2: HSM証明アップロードUI未定義

**問題**: HSM Attestationのアップロード方法が不明確

**解決策**: 
- ファイルアップロードUI追加（セクション6.2.2）
- サポートベンダーのドロップダウン
- リアルタイム検証フィードバック

#### 指摘3: 承認プロセスの可視性

**問題**: 特にCOUNCIL_VOTEモードでの投票状況が見えない

**解決策**:
- 投票進捗バー（セクション6.2.5）
- 残り時間表示
- 投票結果のリアルタイム更新

### 9.2 CIA指摘

#### 指摘1: モード別処理の分離

**問題**: 既存コードでモード別処理が混在

**解決策**: 
- `ProverRegistry.sol`を拡張し、モード別関数を実装
- `EditionConfig`との連携で適切なモードを自動判定

#### 指摘2: HSM検証の未実装

**問題**: `hsmAttestation`を受け取るが検証していない

**解決策**:
- `HSMVerifier.sol`を新規作成（セクション5.1）
- ベンダー別検証ロジック実装
- FIPS Level自動判定

#### 指摘3: Council投票連携

**問題**: Governance契約との連携がない

**解決策**:
- `ProverCouncilVote.sol`を新規作成
- `SecurityCouncil.sol`との連携
- 投票期限管理の自動化
