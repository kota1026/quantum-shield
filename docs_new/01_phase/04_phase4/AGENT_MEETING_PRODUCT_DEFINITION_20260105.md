## 会議の背景

Phase 4でUI/UXを作成し可視化したことで、以下の課題が顕在化：

1. 各プレイヤーの役割定義が曖昧
2. Prover登録要件などの詳細が未定義
3. 個別最適化では手戻りが発生する懸念

これを受け、サービス全体を俯瞰し、体系的に整理するための会議を開催。

### 検討フロー

```
Phase 0: サービスの価値定義（本会議）
    ↓
①: 登場プレイヤーの役割分担構想（本会議）
    ↓
②: プレイヤー要件の詳細定義（次回以降）
    ↓
③: UI/UXへの要件定義
    ↓
④: 全体UI/UXの整合性確認
    ↓
⑤: コンプラ観点での課題洗い出し
    ↓
⑥: 課題解決の方向性策定
```

## 議題1: サービスの価値定義（Why Quantum Shield?）

### ターゲットセグメント分析

| セグメント | 課題 | 既存解決策の限界 | QSの提供価値 |
|-----------|------|-----------------|-------------|
| **機関投資家** | 量子コンピュータによる暗号破壊リスク | 既存ブリッジは量子脆弱 | NIST準拠の量子耐性 |
| **DeFiプロトコル** | ブリッジハック被害（$2B+） | 中央集権的マルチシグ | 分散型Prover + Slashing |
| **企業トレジャリー** | 規制対応・監査証跡 | 不透明なオフチェーン処理 | 完全オンチェーン透明性 |
| **L2/L3プロジェクト** | セキュアなブリッジ構築コスト | 独自開発は高コスト・高リスク | 量子耐性ブリッジ as a Service |

### 収益モデル

## 議題2: プレイヤー役割構想

### プレイヤー一覧（会議前の認識）

| プレイヤー | 現状の定義 | 課題・未定義事項 |
|-----------|----------|-----------------|
| **User** | Lock/Unlockする一般利用者 | 要件は明確 |
| **Prover** | SPHINCS+署名を行う検証者 | ⚠️ 登録要件が曖昧 |
| **Operator** | 未定義 | ⚠️ 役割不明 |
| **Security Council** | 緊急対応・ガバナンス | Phase 2以降 |
| **Guardian** | 未定義（監視者？） | ⚠️ 役割不明 |
| **Challenger** | 不正検知・Challenge提起 | 要件は概ね明確 |
| **監視ボット** | 24/7監視 | 実装詳細未定 |

### 論点1: Prover登録要件（CSO提起）

**現状の仕様書（SEQUENCES.md #5）:**
- 最低Stake: $400K-$500K
- HSM使用: 必須
- 2-of-3マルチシグ: 必須
- 法的契約: 必須

**未定義事項:**
1. 技術審査の具体的プロセスは？
2. HSM使用の証明方法は？
3. 法的契約の締結相手は？（QS財団？）
4. Phase別の承認フローの詳細は？
5. Proverの地理的分散要件は？

### 論点2: Edition別Prover要件（CBO提起）

| 要件 | Enterprise | Decentralized |
|------|-----------|---------------|
| 参入 | 契約ベース（厳格） | Stake + 自動承認 |
| 審査 | QS財団による面談 | 技術要件のみ |
| 責任 | 契約上の賠償責任 | Slashingのみ |
| 報酬 | 固定 + 変動 | Stake比例 |

### 論点3: Operatorの定義（CTO提起）

| 役割 | 責任 | 例 |
|------|-----|-----|
| **L3 Node Operator** | L3ノードの運営 | 4ノードの運営者 |
| **Prover Operator** | Proverインフラ運営 | HSM + 署名システム |
| **Bridge Operator** | ブリッジ全体の運営 | Enterprise版顧客 |

### 論点4: 用語整理（Legal提起）

| 現行用語 | 提案 | 理由 |
|---------|------|------|
| Guardian | **Observer** | 法的責任を示唆しない |
| 監視ボット | **Monitor** | 技術的役割を明確化 |
| Challenger | Challenger（維持） | 法的に問題なし |

### 論点5: インセンティブ設計（CFO提起）

| プレイヤー | インセンティブ | ペナルティ |
|-----------|--------------|-----------|
| User | 安全な資産移動 | N/A |
| Prover | 手数料報酬（50%配分） | Quadratic Slashing |
| Observer | Whistleblower報酬（$100K） | N/A |
| Challenger | Slash額の60% | Bond没収（敗訴時） |
| L3 Node Operator | インフラ報酬（Treasury） | N/A |
| Security Council | 年間$50K報酬 | 任期制限 |

### 論点6: 攻撃ベクトル分析（Red Team提起）

| プレイヤー | 攻撃ベクトル | 対策状況 |
|-----------|------------|---------|
| Prover | 共謀攻撃 | ⚠️ Quadratic Slashingあるが、5社全員共謀は100%Slashでも足りない可能性 |
| Prover | HSM鍵漏洩 | ⚠️ 2-of-3要件あるが、全鍵漏洩時の対策は？ |
| L3 Node | BFT攻撃（2/4共謀） | ⚠️ 4ノード中2ノード掌握で停止可能 |
| Observer | 偽Alert | ✅ Challenge Bondで抑止 |
| Challenger | 悪意のChallenge | ✅ Bond没収で抑止 |
