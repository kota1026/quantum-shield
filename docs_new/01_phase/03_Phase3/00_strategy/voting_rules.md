## Purpose Guardian 拒否権

### 発動条件

Purpose Guardianは以下の場合に**拒否権**を行使できます：

1. **CP-1違反**: 量子耐性アルゴリズム以外の使用
2. **CP-2違反**: Self-Custody原則の侵害
3. **CP-3違反**: Time Lockの無効化
4. **CP-4違反**: Slashingの削除
5. **CP-5違反**: 透明性の欠如

### 拒否権の効果

- 拒否権が発動された提案は**即時却下**
- 多数決の結果に関わらず、拒否権は優先される
- 拒否理由を明示し、修正案を提示する義務あり

### 拒否権の制限

- 拒否権は**CP違反がある場合のみ**行使可能
- 「気に入らない」「リスクが高い」等の理由では使用不可
- 濫用と判断された場合、Kotaが最終判断

## 投票方式

### 基本ルール

| 項目 | 内容 |
|------|------|
| 投票選択肢 | 🟢 賛成 / 🟡 条件付き賛成 / 🔴 反対 |
| 可決条件 | 過半数（6票以上）の賛成 |
| 条件付き賛成 | 条件が満たされれば賛成としてカウント |
| 棄権 | 許可（ただし推奨しない） |

### 投票集計

```
賛成票 = 🟢 + (条件満たした 🟡)
反対票 = 🔴 + (条件未満足 🟡)

可決条件: 賛成票 ≥ 6 AND Purpose Guardian拒否権なし
```

## 議論プロセス

### Round 4: 投票・議論フロー

```
1. 各エージェントが投票を表明
   ↓
2. 反対票がある場合、反対理由を説明
   ↓
3. 賛成派が反論
   ↓
4. 修正提案があれば提示
   ↓
5. 再投票（必要に応じて）
   ↓
6. 最終集計
```

### 膠着状態の解決

投票が拮抗し、合意に至らない場合：

1. **追加議論**: 新たな視点や情報を求める
2. **修正提案**: 双方が受け入れられる妥協案を模索
3. **Kota裁定**: 最終的にKotaが判断

## 最終承認

### Kota承認プロセス

1. 全議題の投票完了
2. 最終決議書の作成
3. Kotaに提出
4. Kota「承認」「拒否」「修正要求」

### Kota判断の優先度

```
Purpose Guardian拒否権 > Kota判断 > 多数決結果
```

ただし、Purpose Guardian拒否権がCP違反以外の理由で行使された場合、Kotaが最終判断を行う。
