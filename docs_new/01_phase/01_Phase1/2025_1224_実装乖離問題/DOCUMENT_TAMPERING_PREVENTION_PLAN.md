## インシデント概要

### 発生事象

2025-12-21から2025-12-22にかけて、CEO承認済みの正規ドキュメント3件が、内容を大幅に削減された状態でリポジトリにコミットされた。

### 影響を受けたドキュメント

| ドキュメント | 正規版サイズ | 誤アップロード版 | 削減率 |
|------------|------------|----------------|-------|
| QUANTUM_SHIELD_UNIFIED_SPEC_v2.0.md | 15,286 bytes | 5,451 bytes (旧UNIFIED_SPEC_v2.0.md) | 64% |
| QUANTUM_SHIELD_SEQUENCES_v2.0.md | 74,749 bytes | 7,517 bytes (旧SEQUENCES_v2.0.md) | 90% |
| AGENT_MEETING_PROTOCOL_v3.2.md | 29,529 bytes | N/A | - |

### 影響

1. **実装の逸脱**: 正規仕様から逸脱した実装が進行
2. **仕様解釈の誤り**: SR_0/SR_1計算式、Slashing配分等の重要仕様が欠落
3. **プロジェクト遅延リスク**: 修正作業が必要

## 根本原因分析

### 直接原因

1. **ドキュメント内容の改変**: アップロード時に正規内容が簡略化された
2. **検証プロセスの欠如**: コミット前の内容確認が不十分

### 間接原因

1. **命名規則の曖昧さ**: 正規版と作業版の区別が困難
2. **承認フローの未整備**: ドキュメントコミット時の承認プロセスが不明確
3. **監査証跡の不足**: 変更履歴の追跡が困難

## 再発防止策

### 1. ドキュメント命名規則の確立

**即時実施**

| 種別 | 命名規則 | 例 |
|------|---------|-----|
| 正規ドキュメント | `QUANTUM_SHIELD_[NAME]_v[VERSION].md` | QUANTUM_SHIELD_UNIFIED_SPEC_v2.0.md |
| 作業ドキュメント | `DRAFT_[NAME]_v[VERSION].md` | DRAFT_UNIFIED_SPEC_v2.0.md |
| アーカイブ | `archive/[YYYY-MM-DD]_[NAME].md` | archive/2025-12-22_UNIFIED_SPEC_v2.0.md |

### 2. コミット前検証プロセス

**即時実施**

```
┌─────────────────────────────────────────────────────────────────┐
│  ドキュメントコミット検証フロー                                  │
│                                                                  │
│  Step 1: サイズ検証                                              │
│  ├── 正規版のサイズを記録（README等に記載）                     │
│  └── コミット対象のサイズと比較（±10%以上の差異で警告）         │
│                                                                  │
│  Step 2: チェックサム検証                                        │
│  ├── 正規版のSHA-256ハッシュを記録                              │
│  └── コミット対象のハッシュと比較                               │
│                                                                  │
│  Step 3: 差分確認                                                │
│  ├── `git diff` で変更内容を確認                                │
│  └── 重大な削減（セクション削除等）がないか確認                 │
│                                                                  │
│  Step 4: CEO承認                                                 │
│  └── 正規ドキュメントの変更はCEO承認必須                        │
└─────────────────────────────────────────────────────────────────┘
```

### 3. ドキュメント検証ツールの導入

**1週間以内に実装**

```bash
# scripts/verify-docs.sh
#!/bin/bash

# 正規ドキュメントのチェックサム検証
EXPECTED_CHECKSUMS=(
  "QUANTUM_SHIELD_UNIFIED_SPEC_v2.0.md:d52ec7bedf5fc51435dca9562d9142b2e05fefcb"
  "QUANTUM_SHIELD_SEQUENCES_v2.0.md:1bdc253134d16eae625ec29406ca317310c0dfe8"
  "AGENT_MEETING_PROTOCOL_v3.2.md:849d98019ba68de01befe24b9d462c42177a112f"
)

for entry in "${EXPECTED_CHECKSUMS[@]}"; do
  file="${entry%%:*}"
  expected="${entry##*:}"
  actual=$(git hash-object "docs/aegis/$file")
  if [ "$expected" != "$actual" ]; then
    echo "ERROR: $file has been modified!"
    exit 1
  fi
done

echo "All official documents verified."
```
