# Prompt 27: 計画書→タスク抽出プロセス

> **Version**: 1.0
> **Date**: 2026-01-12
> **Purpose**: 計画書から100%カバーのタスクリストを生成する
> **Problem Solved**: 計画66%漏れ問題（PHASE5_INTEGRATION_PLAN → TASK_P5_SUMMARY 事例）

---

## 1. 概要

### 1.1 このPromptが解決する問題

```
┌─────────────────────────────────────────────────────────────────┐
│  問題: 計画書からタスクへの変換で66%が漏れた                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  PHASE5_INTEGRATION_PLAN.md (1100行, 115日)                    │
│          ↓ 不完全な変換                                         │
│  TASK_P5_SUMMARY.md (12タスク, 40日) ← 34%しかカバーしていない  │
│                                                                 │
│  漏れた内容:                                                    │
│  - Phase 5.0 ブロッカー (31日) → 全て漏れ                       │
│  - 管理系API (50 EP) → 全て漏れ                                 │
│  - 統合・テスト → 全て漏れ                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 解決アプローチ: 3フェーズ分離

```
┌─────────────────────────────────────────────────────────────────┐
│  Phase A: 全項目抽出 (Extraction)                               │
│  ├── 計画書の全セクション走査                                   │
│  ├── Appendix の全リスト抽出                                    │
│  └── 数値目標の特定                                             │
├─────────────────────────────────────────────────────────────────┤
│  Phase B: 整合性検証 (Validation)                               │
│  ├── 計画数値 vs タスク数値の比較                               │
│  ├── ブロッカーの優先度確認                                     │
│  └── 欠番チェック                                               │
├─────────────────────────────────────────────────────────────────┤
│  Phase C: タスクリスト生成 (Generation)                         │
│  ├── 依存関係付与                                               │
│  ├── 計画参照紐づけ                                             │
│  └── 完了条件定義                                               │
└─────────────────────────────────────────────────────────────────┘
```

## 2. Phase A: 全項目抽出

### 2.1 必須抽出セクション

計画書を読む際、以下のセクションから**必ず**項目を抽出する：

| セクション種別 | 抽出対象 | 漏れやすさ |
|--------------|----------|:----------:|
| **Executive Summary** | 数値目標（工数、EP数） | 中 |
| **ギャップ一覧** | 「未実装」「❌」マーク項目 | 高 |
| **Phase X.0（ブロッカー）** | 全項目 | **極高** |
| **詳細セクション** | 実装タスク | 中 |
| **Appendix** | API一覧、ファイル一覧 | **極高** |
| **完了基準** | 各チェック項目 | 高 |

### 2.2 抽出テンプレート

計画書から以下の形式で抽出する：

```markdown
## 抽出結果

### 数値目標
- 総工数: {N}日
- API EP数: {M} EP
- コントラクト数: {K}
- フェーズ数: {P}

### ブロッカー項目（Phase X.0）
| 項目 | 計画参照 | 工数 | 優先度 |
|------|---------|:----:|:------:|
| {項目1} | §{参照} | {N}日 | P0 |
| ... | | | |

### 通常タスク
| 項目 | 計画参照 | 工数 | 優先度 |
|------|---------|:----:|:------:|
| {項目1} | §{参照} | {N}日 | P1 |
| ... | | | |

### Appendix項目
| リスト名 | 項目数 | 対応タスク必要 |
|---------|:-----:|:--------------:|
| API一覧 | {N} EP | {Y/N} |
| ファイル一覧 | {N}ファイル | {Y/N} |
| 完了基準 | {N}項目 | {Y/N} |
```

### 2.3 抽出チェックリスト

```
□ 計画書の全セクション見出しを列挙したか?
□ 「Phase X.0」「ブロッカー」「P0」「最優先」を検索したか?
□ 全Appendixの項目数をカウントしたか?
□ 「❌」「未実装」「TODO」「STUB」を検索したか?
□ 完了基準セクションを読んだか?
```

## 3. Phase B: 整合性検証

### 3.1 数値整合性チェック

```
┌─────────────────────────────────────────────────────────────────┐
│  チェック1: 工数合計                                            │
│  計画総工数: {plan_days}日                                      │
│  タスク総工数: {task_days}日                                    │
│  乖離: {diff}日 ({diff_pct}%)                                   │
│  判定: {diff_pct} <= 10% ? ✅ PASS : ❌ FAIL                    │
├─────────────────────────────────────────────────────────────────┤
│  チェック2: API EP数                                            │
│  計画API: {plan_ep} EP                                          │
│  タスクAPI: {task_ep} EP                                        │
│  乖離: {diff} EP                                                │
│  判定: {diff} == 0 ? ✅ PASS : ❌ FAIL                          │
├─────────────────────────────────────────────────────────────────┤
│  チェック3: ブロッカーカバー率                                  │
│  計画ブロッカー: {plan_blockers}                                │
│  タスクブロッカー: {task_blockers}                              │
│  判定: {task_blockers} >= {plan_blockers} ? ✅ PASS : ❌ FAIL  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 優先度検証

```python
# 疑似コード
def validate_priority(tasks):
    blockers = [t for t in tasks if t.priority == "P0"]
    if len(blockers) == 0:
        raise Error("P0ブロッカーが0件 - 計画書Phase X.0を確認")

    for blocker in blockers:
        if blocker.plan_ref is None:
            raise Error(f"{blocker.id}: 計画参照がない")

    return True
```

### 3.3 番号連続性チェック

```
タスク番号: P5-001, 002, 003, ..., 035
欠番検出: なし → ✅ PASS
欠番検出: あり → ❌ FAIL（欠番を埋めるか、理由を記載）
```

## 4. Phase C: タスクリスト生成

### 4.1 タスクフォーマット

```markdown
| Task ID | 内容 | 工数 | 依存 | 計画参照 | 状態 |
|---------|------|:----:|------|---------|:----:|
| TASK-P{phase}-{seq} | {description} | {days}日 | {deps} | §{ref} | [ ] |
```

### 4.2 必須フィールド

| フィールド | 説明 | 必須 |
|-----------|------|:----:|
| Task ID | 一意識別子（連番） | ✅ |
| 内容 | タスク説明 | ✅ |
| 工数 | 見積もり日数 | ✅ |
| 依存 | 先行タスクID | ⚪ |
| 計画参照 | 計画書のセクション番号 | ✅ |
| 状態 | [ ] or [x] | ✅ |

### 4.3 依存関係ルール

```
1. Phase X.0タスク → 依存なしで最初に配置
2. API実装タスク → 認証タスク(auth)に依存
3. 統合テスト → 全実装タスクに依存
4. デプロイ → 全テストに依存
```

### 4.4 完了条件テンプレート

各Phaseに完了条件を付与：

```markdown
### Phase X.Y 完了条件

```
[ ] {条件1}
[ ] {条件2}
[ ] {条件3}
```
```

## 5. 実行手順

### Step 1: 計画書読み込み

```
1. 計画書を全文読む
2. セクション構造を把握
3. 数値目標を抽出（工数、EP数）
```

### Step 2: 抽出実行

```
1. Phase X.0（ブロッカー）を最初に抽出
2. 各Phaseのタスクを抽出
3. Appendixの全リストを確認
4. 完了基準をタスクに変換
```

### Step 3: 整合性検証

```
1. 数値チェック実行
2. 優先度チェック実行
3. 番号チェック実行
4. FAIL時は抽出をやり直し
```

### Step 4: タスクリスト生成

```
1. フォーマットに従ってタスク定義
2. 依存関係を付与
3. 完了条件を追加
4. 最終レビュー
```

## 6. アンチパターン（避けるべき）

### 6.1 ❌ ブロッカースキップ

```
問題: Phase X.0を読み飛ばす
結果: 最重要タスクが漏れる
対策: 「Phase 0」「ブロッカー」「P0」を必ず検索
```

### 6.2 ❌ Appendix無視

```
問題: 本文のみ読んでAppendixを飛ばす
結果: API一覧、ファイル一覧が漏れる
対策: Appendixを最初に確認し、項目数をカウント
```

### 6.3 ❌ 数値未検証

```
問題: 計画の数値とタスクの数値を比較しない
結果: 66%漏れに気づかない
対策: 必ず数値整合性チェックを実行
```

### 6.4 ❌ 欠番放置

```
問題: タスク番号に欠番があっても無視
結果: 漏れタスクが見えない
対策: 番号連続性チェック、欠番理由の記載
```

## 7. 成功基準

タスクリスト生成完了時、以下が全て✅であること：

```
□ 計画工数 ≈ タスク工数（±10%）
□ 計画API EP = タスクAPI EP
□ Phase X.0 ブロッカー 100%カバー
□ Appendix 100%カバー
□ タスク番号に欠番なし
□ 全タスクに計画参照あり
□ 全タスクに完了条件あり
```
