## 重要な設計判断と理由

### 1. なぜ Dilithium + SPHINCS+ の二重署名？

**判断**: 量子耐性のために二つの署名アルゴリズムを併用

**理由**:
- Dilithium: NIST標準、高速だがまだ新しい
- SPHINCS+: ハッシュベースで安全性が高い、バックアップとして
- 両方を使うことで、一方が破られても安全

### 2. なぜ 24時間待機期間？

**判断**: 通常アンロックに24時間の待機期間を設ける

**理由**:
- 不正検知の時間を確保
- Observerがチャレンジする猶予
- ユーザーの誤操作を取り消すチャンス

### 3. なぜ L1 + L3 の二層構造？

**判断**: Ethereum (L1) + 独自L3チェーンの構成

**理由**:
- L1: 最終的な資産の安全性を担保
- L3: 高速な署名処理、低コスト
- L2をスキップして直接L3を構築

### 4. なぜ Japan Premium デザイン？

**判断**: 日の丸レッド + ゴールドをブランドカラーに

**理由**:
- 日本発のプロジェクトとしてのアイデンティティ
- 「守る」「安心」のイメージ
- 高級感・信頼感の演出

## 失敗と学び

### 学び1: ドキュメントは実装前に検証すべき

**問題**: 仕様書を書いても、実装時に不足が判明

**解決**: Lock機能でミニ検証を実施し、不足点を洗い出し

**結果**: IMPLEMENTATION_GUIDE.md に学びを反映

### 学び2: 既存コードを先に確認すべき

**問題**: 新規仕様を書いたが、既に実装があった

**解決**: コードベースを先に調査してから仕様を書く

**結果**: CODEBASE_MAP.md を作成

### 学び3: Mock APIから始めるべき

**問題**: UI実装とAPI実装が並行せず、結合で問題発生

**解決**: Mock APIを先に作成し、UI開発を進められるように

**結果**: 開発フローに Mock API 作成を組み込み
