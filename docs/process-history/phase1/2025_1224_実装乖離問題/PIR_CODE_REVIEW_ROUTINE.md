# PIR Code Review Routine v1.0

> **Purpose**: 実装・テスト完了後に必ず実行するコードレベルレビュールーティン
> **Trigger**: 各PIR会議の開始時に本ドキュメントを参照すること

---

## 🔄 ルーティンフロー

```
実装完了 → テスト実行(全PASS) → [本ルーティン開始] → PIR会議 → 判定
```

---

## ✅ 必須チェックリスト

### Phase 1: コード取得・レビュー

| # | 項目 | 確認方法 | 完了 |
|---|------|----------|------|
| 1.1 | 実装コード取得 | `github:get_file_contents` で対象ファイル取得 | ☐ |
| 1.2 | テストコード取得 | `github:get_file_contents` でテストファイル取得 | ☐ |
| 1.3 | 仕様書参照 | 該当仕様書を開き、実装との照合準備 | ☐ |

### Phase 2: 実装コードレビュー

| # | 項目 | チェック内容 | 完了 |
|---|------|--------------|------|
| 2.1 | 仕様準拠 | 関数シグネチャ、パラメータ順序、計算式が仕様書と一致 | ☐ |
| 2.2 | セキュリティ | ReentrancyGuard, アクセス制御, オーバーフロー対策 | ☐ |
| 2.3 | イベント発行 | 必要なイベントが正しいパラメータで発行される | ☐ |
| 2.4 | エラー処理 | 適切なrevert条件とカスタムエラー | ☐ |
| 2.5 | ガス効率 | 不要なストレージ読み書き、ループ内のSLOAD | ☐ |

### Phase 3: テストコードレビュー

| # | 項目 | チェック内容 | 完了 |
|---|------|--------------|------|
| 3.1 | 正常系テスト | 主要機能の成功パスがテストされている | ☐ |
| 3.2 | 異常系テスト | エラーケース(revert)がテストされている | ☐ |
| 3.3 | 境界値テスト | 0, max, 境界条件がテストされている | ☐ |
| 3.4 | イベント検証 | `vm.expectEmit`で全パラメータが検証されている | ☐ |
| 3.5 | 状態変化検証 | 関数実行前後の状態変化が検証されている | ☐ |
| 3.6 | Fuzzテスト | 重要な計算にFuzzテストがある | ☐ |

### Phase 4: 11エージェントレビュー

各エージェントが専門視点でコードを確認:

| Agent | 確認観点 |
|-------|----------|
| Purpose Guardian | Core Principles違反がないか |
| CTO | アーキテクチャ整合性 |
| CSO | セキュリティ脆弱性 |
| CFO | ガスコスト妥当性 |
| CBO | ロードマップ整合性 |
| Engineer | コード品質、可読性 |
| Crypto Auditor | 暗号実装の正確性 |
| Red Team | 攻撃ベクトル |
| QA | テストカバレッジ |
| DevOps | ビルド・デプロイ影響 |
| Legal | コンプライアンス |

## 📋 発見問題の分類

| 重大度 | 定義 | 対応 |
|--------|------|------|
| 🔴 Critical | セキュリティ脆弱性、仕様違反 | 即時修正、PIR FAIL |
| 🟡 Major | テスト不足、品質問題 | 修正後に再レビュー |
| 🟢 Minor | 改善推奨、将来対応 | 記録して次回対応 |

---

## 📝 PIR判定基準

### ✅ PASS条件
- Phase 1-4の全チェック完了
- 🔴 Critical問題: 0件
- 🟡 Major問題: 0件（または修正済み）

### ⚠️ CONDITIONAL PASS条件
- 🟡 Major問題: 1-2件（次PIRまでに修正予定）

### ❌ FAIL条件
- 🔴 Critical問題: 1件以上
- 🟡 Major問題: 3件以上

## 🚀 使用方法

1. PIR開始時に本ドキュメントを`github:get_file_contents`で取得
2. チェックリストを順番に実行

4. 11エージェントレビュー実施
5. 判定結果をPIR議事録に記載
6. WBS更新
