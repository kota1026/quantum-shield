### 3.3 API呼び出しフロー定義

```typescript
// 標準的なAPI呼び出しパターン（全画面で統一）

interface StandardApiFlow {
  // 1. 画面状態
  state: {
    isLoading: boolean;
    error: string | null;
    data: T | null;
  };

  // 2. API呼び出しハンドラ
  handleAction: async () => {
    setIsLoading(true);
    setError(null);

    try {
      const result = await apiCall();
      setData(result);
      showSuccessToast();
      navigateIfNeeded();
    } catch (e) {
      setError(e.message);
      showErrorToast();
    } finally {
      setIsLoading(false);
    }
  };

  // 3. UI表示
  render: {
    loading: <Spinner />,
    error: <ErrorMessage message={error} retry={handleAction} />,
    success: <SuccessContent data={data} />,
  };
}
```

### 4.2 デザインで発生している問題パターン

```
【問題1: 情報密度が高すぎて見づらい】

原因:
- 1画面に詰め込みすぎ
- 視覚的階層がない

解決策:
- 最大3階層の視覚的ヒエラルキー
- KPIは最大6個まで
- テーブルは10列まで
- 折りたたみ可能なセクション
```

```
【問題2: ヘッダーの渋滞】

原因:
- ロゴ、ナビ、環境セレクター、検索、通知、ユーザーメニューが1行に密集

解決策:
- デスクトップ: 左(ロゴ+メインナビ) / 中央(検索) / 右(通知+ユーザー)
- 環境セレクターは左上に独立配置（カラーバーで視覚的に区別）
- モバイル: ハンバーガーメニュー + アイコンのみ
```

```
【問題3: 何をする画面か分からない】

原因:
- タイトルが不明確
- 説明文がない
- CTAが目立たない

解決策:
- 全画面に明確なタイトル + サブタイトル（説明）
- メインCTAは画面に1つ（primary variant）
- 空状態の説明とアクション誘導
```

### 4.3 プロンプト改善案

```markdown
# 実装時プロンプト改善案

## デザイン確認チェックリスト（必須）

実装前に以下を確認:

□ DESIGN_SYSTEM.md に定義されたコンポーネントのみ使用
□ 定義されていないパターンは実装前に DESIGN_SYSTEM.md に追加
□ カスタムスタイル（カラー、スペーシング）を使用しない
□ 1画面に primary ボタンは1つのみ
□ 情報階層: 見出し(2xl) → サブ見出し(lg) → 本文(base)
□ タップエリア: 最小44px × 44px

## ユーザーアクション完了チェックリスト（必須）

ボタン/リンク実装時に以下を定義:

□ クリック後のローディング状態
□ 成功時の遷移先 or UI更新
□ 成功時のフィードバック（トースト、モーダル）
□ エラー時の表示
□ エラーからの復帰方法（リトライ、戻る）

## 画面遷移チェックリスト（必須）

新規画面実装時に以下を明確化:

□ この画面への全ての入り口（どこから来るか）
□ この画面からの全ての出口（どこへ行くか）
□ 「戻る」ボタンの遷移先
□ エラー時のフォールバック画面
□ URL直接アクセス時の挙動
```

## 8. 自動修正方法論

### 8.1 AIエージェントによる自動検証

```typescript
// 自動検証エージェントの構成

interface AutoValidationAgent {
  // 1. ジャーニー整合性チェック
  journeyValidator: {
    input: 'APP_DESIGN_SPECS.md',
    checks: [
      '全画面に入り口と出口が定義されているか',
      '戻るボタンの遷移先が明確か',
      '完了後の遷移が定義されているか',
    ],
    output: 'validation_report.md',
  };

  // 2. ドキュメント整合性チェック
  docConsistencyChecker: {
    inputs: ['APP_DESIGN_SPECS.md', 'URL_REFERENCE.md', 'DATA_MODEL.md'],
    checks: [
      '画面数が一致しているか',
      'エンティティが全て定義されているか',
      'APIエンドポイントがData Modelと整合しているか',
    ],
    output: 'inconsistency_report.md',
  };

  // 3. デザイン準拠チェック
  designComplianceChecker: {
    inputs: ['コンポーネントコード', 'DESIGN_SYSTEM.md'],
    checks: [
      '未定義のカラーを使用していないか',
      '未定義のスペーシングを使用していないか',
      'primaryボタンが1画面に1つか',
    ],
    output: 'design_violations.md',
  };
}
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    自動修正フロー                                 │

│                                                                 │

│     ├─ ドキュメント不整合 → 自動修正可能                         │

│     └─ ジャーニー不整合 → 手動修正（設計判断必要）               │
│                                                                 │
│  4. 自動修正実行                                                 │
│     ├─ PR作成                                                   │
│     └─ レビュー依頼                                              │
│                                                                 │
│  5. 手動修正項目のチケット化                                      │
│     └─ 優先度に応じてバックログ登録                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 8.3 CI/CD統合

```yaml
# .github/workflows/design-validation.yml

name: Design & Journey Validation

on:
  pull_request:
    paths:
      - 'docs/specs/**'
      - 'apps/web/src/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run Journey Validator
        run: node scripts/validate-journey.js

      - name: Run Doc Consistency Checker
        run: node scripts/check-doc-consistency.js

      - name: Run Design Compliance Checker
        run: node scripts/check-design-compliance.js

      - name: Comment on PR
        if: failure()
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              body: 'Design/Journey validation failed. See details above.'
            })
```
