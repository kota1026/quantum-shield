# L3基盤層 技術選定会議 決議記録

**日時**: 2025-12-28  
**議題**: L3の「記録を残す部分」＝ブロックチェーン基盤をどうするか？  
**ステータス**: ✅ 承認済み

---

## 1. 背景

Phase 3.1の進行中に、L3基盤層（ブロックチェーンインフラ）の技術選定が未決定であることが判明。アプリケーション層（l3-aegis）の開発は進んでいたが、以下が未定義だった：

- 合意アルゴリズム（BFT? CometBFT? PBFT?）
- P2Pネットワーキング
- ブロック生成
- 状態永続化
- ノード運用

この状態で設計を続けると、後でインフラ層との不整合が生じるリスクがあった。

## 2. 検討した選択肢

### 2.1 初期選択肢一覧

| # | 選択肢 | 概要 |
|---|--------|------|
| A | 独自ブロックチェーン構築 | P2P、合意、ブロック生成を全部自作 |
| B | Cosmos SDK / CometBFT | ブロックチェーンフレームワークを利用 |
| C | Substrate (Polkadot) | Rust製フレームワーク |
| D | OP Stack Fork | Optimismのスタックを改造 |
| E | Arbitrum Orbit | Arbitrumのスタックを利用 |
| F | オフチェーンサービス + L1アンカー | ブロックチェーンを作らない |
| G | RaaS (Conduit等) | Rollup as a Service |
| H | Sovereign Rollup (Celestia) | データ可用性を分離 |
| I | Based Rollup | L1 Sequencerを使う |
| J | AVS (EigenLayer) | バリデータネットワークを借りる |

### 2.2 CP-1（完全量子耐性）による除外

| 選択肢 | 除外理由 |
|--------|---------|
| G. RaaS | CP-1対応不可、ベンダー依存 |
| I. Based Rollup | L1 ECDSA依存でCP-1違反 |
| J. AVS | BLS署名でCP-1違反 |

## 3. 核心的な議論：オフチェーン vs オンチェーン

### 3.1 シーケンス図による検証

Unlock Sequence (QUANTUM_SHIELD_SEQUENCES_v2.0.md) を分析：

```
User                 L3 Aegis           Chainlink VRF      Prover (5社)        L1 Vault
  │                      │                   │                  │                  │
  │──(1) Unlock Req─────►│                   │                  │                  │
  │                      │                   │                  │                  │
  │                      │──(2) VRF要求─────►│                  │                  │
  │                      │◄──(3) VRF seed───│                  │                  │
  │                      │                   │                  │                  │
  │                      │──(4) 署名要求────────────────────────►│                  │
  │                      │◄──(5) SPHINCS+署名×2 ←←←←←←←←←←←←←←│ 【問題箇所】      │
  │                      │                   │                  │                  │
  │                      │──(6) Submit───────────────────────────────────────────►│
```

**問題箇所**: ステップ(5)「Prover → L3」の通信

### 3.2 オフチェーン通信の問題点

| 問題 | 説明 |
|------|------|
| **否認問題** | Proverが署名を送ったのに「送ってない」と言われる |
| **検閲問題** | L3が署名を受け取ったのに「受け取ってない」と偽る |
| **改ざん問題** | 中間者が署名を改ざん |

### 3.3 攻撃シナリオ

**シナリオ1: L3オペレーター悪意**
```
1. Proverが署名をオフチェーンで送信
2. L3オペレーターが署名を破棄
3. 「署名が来なかった」と主張
4. Emergency Path強制（7日遅延）
5. その間にETH価格操作
```

**シナリオ2: Prover悪意**
```
1. Proverが署名を送らない
2. 後から「送った」と主張
3. 証拠がないので水掛け論
```

**チェーンなら両方防げる** - 全てオンチェーンに記録されるため。

### 3.4 設計変更の影響

オフチェーン→オンチェーン移行で変更が必要な箇所：

1. `IProver` インターフェース → トランザクション形式に変更
2. `IL3Bridge` → ブロック参照に変更
3. 署名収集ロジック → コンセンサス依存に変更
4. タイムアウト処理 → ブロック高さ基準に変更

**結論**: 後から変えるのは大変。最初からチェーンとして設計すべき。

## 4. コスト・性能への影響

### 4.1 ガス代比較

| 項目 | L3 Gas | L3 Cost | L1 Gas | L1 Cost |
|------|--------|---------|--------|---------|
| Unlock Tx (User) | ~100K | ~$0.10 | - | - |
| Sign Tx (Prover×2) | ~50K×2 | ~$0.10 | - | - |
| L1 Submit | - | - | ~300K | ~$30 |
| **合計** | | **~$0.20** | | **~$30** |

**L3のガス代は誤差レベル** (L1が支配的)

### 4.2 証明時間比較

| 方式 | 合計時間 |
|------|---------|
| オフチェーン | ~35秒 |
| オンチェーン | ~47秒 |

**差は12秒程度** - 許容範囲

## 8. 決議

### 決議1: L3は「チェーン」として構築する ✅ 承認

**理由**:
1. Prover署名送信が「オンチェーン」でないと否認・改ざん可能
2. オフチェーン→オンチェーン移行は設計し直しになる
3. CP-5（透明性）の完全準拠にはチェーンが必要
4. ガス代・証明時間への影響は軽微
5. 長期的なナラティブ・ビジネス価値が高い
