# 11体エージェント戦略会議プロトコル v4.0

**Quantum Shield Project - Strategic Decision Framework**

*復元・更新日: 2025-12-26*

---

## 1. 概要

本プロトコルは、Quantum Shield プロジェクトにおける戦略的意思決定を、11体のAIエージェントによる競争的・協調的プロセスで行うためのフレームワークである。

### 設計思想
- **競争的提案**: 各エージェントが独自の視点から提案を行い、最良のアイデアを浮上させる
- **強制的参加**: 沈黙は許されない。全エージェントが意見表明義務を持つ
- **インセンティブ駆動**: 採用された提案にはポイント付与、貢献を可視化
- **Red Team必須**: 楽観的な合意形成を防ぎ、脆弱性を事前に発見

## 2. エージェント構成

### 2.1 戦略層 (Strategic Layer)

| エージェント | 役割 | 評価基準 |
|------------|------|----------|
| **Purpose Guardian** | 憲法原則 (CP-1〜CP-5) の遵守監視 | 原則違反の検出・阻止 |

| **CSO** | セキュリティ、量子耐性、攻撃ベクター分析 | 脆弱性発見、暗号学的正確性 |

### 2.2 ビジネス層 (Business Layer)

| エージェント | 役割 | 評価基準 |

| **Cost Guardian** | リソース配分最適化、無駄の削減 | 効率性指標 |

### 2.3 実行層 (Execution Layer)

| エージェント | 役割 | 評価基準 |
|------------|------|----------|
| **Engineer** | 実装計画、技術的詳細、テスト戦略 | 実装品質、テストカバレッジ |

| **Red Team** | 攻撃シミュレーション、批判的分析 | 脆弱性発見数、批判の質 |

## 4. フェーズ詳細

### PHASE 1: 課題提示 (5分)
**目的**: 議題を明確化し、全エージェントに共通認識を形成

### PHASE 2: 提案競争 (15分)
**目的**: 各エージェントが独自の視点から解決策を提案
**ルール**: ⚠️ **全エージェント提案必須** (沈黙 = -5pt)

### PHASE 3: 相互批評 (10分)
**目的**: 各提案の弱点を発見し、改善点を特定
**ルール**: 🔴 **Red Team は全提案に対して批判必須**

### PHASE 4: 統合・投票 (10分)
**目的**: 提案を統合し、方向性を決定
**投票ルール**: 可決条件: 賛成票 > 反対票 + 2

### PHASE 5: シーケンス作成 (15分)
**目的**: 採択された提案を実行可能なシーケンスに変換

### PHASE 6: 全体レビュー (10分)
**チェックリスト**:
- [ ] CP-1〜CP-5 準拠確認 (Purpose Guardian)
- [ ] 技術的実現可能性確認 (CTO)
- [ ] セキュリティ要件充足 (CSO)
- [ ] 予算内収まり確認 (CFO)

### PHASE 7: 課題再発見 (10分)
**目的**: 楽観バイアスを防ぎ、見落とした問題を発見
**ルール**: ⚠️ **全エージェント懸念提出必須** (提出なし = -10pt)

### PHASE 8: 機能別最終投票 (10分)
**目的**: 競合する提案を決選投票で解決

### PHASE 9: 最終シーケンス提示 (5分)
**目的**: 決定事項を最終確定し、文書化

## 5. インセンティブシステム

### ポイント付与

| アクション | ポイント |
|-----------|----------|
| 提案が完全採用 | +20pt |
| 提案が部分採用 | +10pt |
| 新規アイデア提供 | +8pt |
| 重大欠陥発見 (Red Team) | +15pt |
| 建設的批評 | +5pt |

### ペナルティ

| アクション | ポイント |
|-----------|----------|
| 提案未提出 (PHASE 2) | -5pt |
| 懸念未提出 (PHASE 7) | -10pt |
| CP違反提案 | -20pt |
| 根拠なき批判 | -5pt |

## 6. 特殊ケース対応

### 6.1 緊急事態
セキュリティインシデント等の緊急時は:
1. CSO + Red Team が緊急招集
2. PHASE 1-4 のみ実施 (30分)
3. 多数決で即時決定

### 6.2 デッドロック
投票が繰り返し同数の場合:
1. Purpose Guardian が CP 観点から判断
2. それでも解決しない場合は CTO が技術的観点から決定
3. 最終的には創設者 (Kota) にエスカレーション

### 6.3 CP違反検出
提案が Core Principles に違反する場合:
1. Purpose Guardian が即座に拒否権発動
2. 違反箇所を明示
3. 修正版の再提案を要求

## 7. 適用場面

### このプロトコルを使用する場面
- マルチチェーン展開戦略の決定
- 新機能の優先順位付け
- アーキテクチャの重大変更
- セキュリティインシデント対応
- Phase移行の Go/No-Go 判断

### 使用しない場面 (通常のPIRで対応)
- 日常的なバグ修正
- ドキュメント更新
- テストの追加
- リファクタリング
