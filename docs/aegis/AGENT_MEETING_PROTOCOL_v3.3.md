# Agent Meeting Protocol v3.3

> **Document Version**: 3.3  
> **Last Updated**: 2025-12-22  
> **Purpose**: 11エージェントによる設計・実装会議の標準プロセス

---

## Overview

Agent Meeting Protocol は、11人のAIエージェントが協力して複雑な設計課題を解決するための構造化されたプロセスである。各エージェントは専門的な視点を持ち、提案、批評、投票を通じて最適な設計・実装に到達する。

### Key Features

- **強制懸念提出**: 各エージェントが最低1つの懸念を提出必須
- **機能別投票**: 全機能を個別に投票で確定
- **イテレーション**: 重大懸念は追加Roundで解決
- **外部レビュー統合**: ChatGPT/Gemini等の外部AIの批評を統合可能
- **🆕 実装フェーズ対応**: AIP (Agent Implementation Protocol) による品質保証

### v3.3 新機能

| 機能 | 説明 |
|------|------|
| AIP統合 | 実装フェーズ専用プロトコルを追加 |
| Design vs Implementation | フェーズに応じた会議タイプの使い分け |
| 実装チェックリスト | Security / Performance / Crypto / Purpose |
| テスト戦略必須化 | テストなき実装は禁止 |

## Agent Roster

### Strategic Layer（戦略層）

| Agent | Role | Design Focus | Implementation Focus |
|-------|------|-------------|---------------------|
| 🛡️ Purpose Guardian | 理念の守護者 | Core Principlesとの整合性 | **[Gatekeeper]** 実装でも原則遵守確認 |
| 🔧 CTO | 技術統括 | アーキテクチャ、実装可能性 | 技術的負債監視、コード品質基準 |
| 🔐 CSO | セキュリティ統括 | 攻撃ベクトル、リスク評価 | 脆弱性チェック、依存関係リスク |

### Business Layer（ビジネス層）

| Agent | Role | Design Focus | Implementation Focus |
|-------|------|-------------|---------------------|
| 💰 CFO | 財務統括 | 経済モデル、収益性 | コスト対効果確認 |
| 📈 CBO | 事業開発 | 市場、競合、パートナーシップ | UXへの影響確認 |
| 💵 Cost Guardian | コスト監視 | Gas代、運営コスト、効率性 | Gas最適化、計算量O()監査 |

### Execution Layer（実行層）

| Agent | Role | Design Focus | Implementation Focus |
|-------|------|-------------|---------------------|
| 👨‍💻 Engineer | 実装担当 | コード、インフラ、技術詳細 | **[主担当]** 実装詳細、ライブラリ選定 |
| 🧮 Chief Cryptographer | 暗号専門家 | 暗号アルゴリズム、数学的安全性 | パラメータ正確性、サイドチャネル対策 |
| 📋 Researcher | 研究者 | 学術的根拠、ゲーム理論、形式検証 | 最新攻撃論文との照合 |
| ⚖️ Legal | 法務担当 | 規制、コンプライアンス、契約 | ライセンス互換性確認 |
| 🔴 Red Team | 攻撃者視点 | 攻撃シナリオ、脆弱性発見 | **[最重要]** テストシナリオ作成 |

│  ├── 依存性: 外部システム障害時は？                                        │

│  ├── 「問題なし」は禁止（-10pt）                                           │

# Part 2: Agent Implementation Protocol (AIP) v1.0

> **Purpose**: 実装タスクの具体化、レビュー、品質保証プロセス  
> **Focus**: "Why/What"（設計）から "How"（実装・テスト）への移行  
> **Duration**: 60分上限

## AIP Overview

実装フェーズの会議では、**「動くコード（Running Code）」**と**「堅牢な安全性（Security）」**を最優先事項とします。設計会議のような「拡散的な議論」ではなく、「収束的な品質向上」を目指します。

### Core Shifts (Design vs Implementation)

| 項目 | Design Phase | Implementation Phase |
|------|-------------|---------------------|
| **目的** | 最適なアーキテクチャの決定 | バグのないセキュアな実装 |
| **主役** | CBO / Researcher / CTO | **Engineer / Red Team / Security** |
| **成果物** | 仕様書 / シーケンス図 | **実装設計 / テスト計画** |
| **決定方法** | 投票（多数決） | **承認（Go/No-Go判定）** |

---

## AIP Phase Details

### PHASE 1: Task Definition

**目的**: 「何を作るか」の認識合わせ

**必須項目**:
- チケット/Issue番号
- Input/Outputデータ型
- 制約条件（メモリ、時間、セキュリティレベル）
- 参照すべき正規ドキュメント（QUANTUM_SHIELD_*）

**フォーマット**:
```
┌─────────────────────────────────────────────────────────────────┐
│  Task: [チケット番号] [タスク名]                                │
│                                                                 │
│  Input:                                                         │
│  ├── [入力1]: [型]                                             │
│  └── [入力2]: [型]                                             │
│                                                                 │
│  Output:                                                        │
│  └── [出力]: [型]                                              │
│                                                                 │
│  Constraints:                                                   │
│  ├── Gas: [上限]                                               │
│  ├── Time: [上限]                                              │
│  └── Security: [レベル]                                        │
│                                                                 │
│  Reference:                                                     │
│  └── QUANTUM_SHIELD_[DOC]_v[X.X].md Section [Y]                │
└─────────────────────────────────────────────────────────────────┘
```

### PHASE 2: Tech Draft

**目的**: 「どう作るか」の具体化

**担当**: 👨‍💻 Engineer

**必須項目**:
| 項目 | 説明 | 例 |
|------|------|-----|
| 言語・バージョン | 使用言語とバージョン | Solidity ^0.8.20 |
| ライブラリ | 採用するライブラリ | OpenZeppelin 5.0 |
| データ構造 | Struct/Enum定義 | `struct Lock { ... }` |
| エラーハンドリング | 方針 | Custom errors使用 |
| 擬似コード | 主要ロジック | [コード] |
| 学術的根拠 | 参照論文/標準 | FIPS 204, NIST SP 800-185 |

**擬似コードレビュー**:
- 全エージェントがコメント可能（10分上限）
- 深刻な問題は次フェーズで対応

### PHASE 3: Domain Audit

**目的**: 分野別の詳細チェック

#### 3.1 Purpose Check (🛡️ Purpose Guardian)

**チェック項目** (Critical):
- [ ] 量子耐性: NIST準拠アルゴリズムか？
- [ ] Self-Custody: ユーザーが秘密鍵を管理しているか？
- [ ] Time Lock: ゼロ不可のTime Lockが存在するか？
- [ ] Slashing: 削除不可能なSlashingが存在するか？
- [ ] 透明性: 全てオンチェーン検証可能か？

**🔴 違反発見時は即時No-Go**

#### 3.2 Security Check (🔐 CSO / 🔴 Red Team)

**チェック項目**:
| 優先度 | 項目 | 説明 |
|--------|------|------|
| Critical | Input Validation | 全外部入力は検証されているか？ |
| Critical | Access Control | 関数呼び出し権限は適切か？ |
| Critical | Dependencies | 使用ライブラリに既知の脆弱性はないか？(`cargo audit`) |
| Major | Secrets | 秘密鍵やAPIキーがログ出力されないか？ |
| Major | Fail-Safe | エラー時にシステムは安全な状態で停止するか？ |
| Minor | Reentrancy | 再入攻撃対策は適切か？ |

#### 3.3 Performance Check (💵 Cost Guardian / 🔧 CTO)

**チェック項目**:
| 優先度 | 項目 | 説明 |
|--------|------|------|
| Critical | Benchmark | Gas/時間/メモリの3指標を計測したか？ |
| Critical | Storage | 不必要なSSTOREを行っていないか？ |
| Major | Memory | 大きな配列のコピーを避けているか？ |
| Major | Loop | 無限ループやGas Limit超過のリスクはないか？ |
| Minor | Optimization | 最適化の余地はあるか？ |

**ベンチマーク閾値**:
- 同等機能の既存実装と比較して+20%以内
- 絶対値での上限も考慮

#### 3.4 Crypto Check (🧮 Chief Cryptographer)

**チェック項目**:
| 優先度 | 項目 | 説明 |
|--------|------|------|
| Critical | Algorithm | NIST準拠（FIPS 204/205/202）のパラメータか？ |
| Critical | Constant Time | 定数時間実装か？（タイミング攻撃対策） |
| Critical | Entropy | 乱数源は暗号論的に安全か？ |
| Major | Side Channel | サイドチャネル攻撃対策は十分か？ |
| Major | Key Management | 鍵管理フローは適切か？ |
| Minor | Agility | 将来アルゴリズム交換可能な設計か？ |

#### 3.5 License Check (⚖️ Legal)

**チェック項目**:
| 優先度 | 項目 | 説明 |
|--------|------|------|
| Critical | GPL Contamination | GPL/LGPLライブラリが混入していないか？ |
| Major | License Compatibility | MIT/Apache互換のライセンスか？ |
| Minor | Attribution | 必要な帰属表示はあるか？ |

### PHASE 4: Test Strategy

**目的**: 実装と対になるテスト計画の策定

**原則**: 「テストのない実装は認めない」

**担当**: 🔴 Red Team (Lead), 👨‍💻 Engineer

**出力テンプレート**:

```
┌─────────────────────────────────────────────────────────────────┐
│  Test Strategy: [タスク名]                                      │
│                                                                 │
│  ✅ Happy Path (正常系):                                        │
│  ├── [テスト1]: 期待通り動作するか                             │
│  └── [テスト2]: 正常な入力で正常な出力か                       │
│                                                                 │
│  ⚠️ Error Path (異常系):                                        │
│  ├── [テスト3]: 不正入力でエラーを返すか                       │
│  └── [テスト4]: 権限なしでrevertするか                         │
│                                                                 │
│  💣 Edge Cases (境界値):                                        │
│  ├── [テスト5]: 空データ                                       │
│  ├── [テスト6]: 最大データ                                     │
│  └── [テスト7]: 境界値（0, MAX_UINT256等）                     │
│                                                                 │
│  🔨 Attack Vectors (攻撃シナリオ) - 5カテゴリ必須:             │
│  ├── 1. 入力バリデーション: [シナリオ]                         │
│  ├── 2. 状態操作: [シナリオ]                                   │
│  ├── 3. タイミング: [シナリオ]                                 │
│  ├── 4. 経済的: [シナリオ]                                     │
│  └── 5. 依存関係: [シナリオ]                                   │
│                                                                 │
│  🧪 Test Types:                                                 │
│  ├── Unit Test: [カバレッジ目標]                               │
│  ├── Fuzz Test: [実行回数]                                     │
│  ├── Integration Test: [対象]                                  │
│  └── Mock: [HSM等環境依存部分]                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**攻撃シナリオローテーション制**:
- 毎回同じシナリオの形骸化を防ぐため、カテゴリ内で異なる攻撃手法を選択

### PHASE 5: Forced Concern

**目的**: 集団思考の防止

**ルール**:
- 各エージェントは実装に関する懸念を最低1つ提出
- 「特になし」は禁止（-10pt）
- 的外れな懸念でも減点なし（萎縮防止）
- 懸念点が後で実問題になった場合: +15pt

**分類**:
| レベル | 対応 |
|--------|------|
| Critical | 即時対応必須（No-Go要因） |
| Major | 対応推奨（実装前に解決すべき） |
| Minor | 認識のみ（リスクレジストリに記載） |

### PHASE 6: Go / No-Go Decision

**目的**: 実装着手の承認

**No-Go条件（1つでも該当すれば却下）**:
1. 🔴 Core Principles違反
2. 🔴 Critical懸念が未解決
3. 🔴 セキュリティテスト計画が不十分
4. 🔴 ベンチマーク未実施
5. 🔴 依存ライブラリに既知の脆弱性

**Go条件**:
- 上記No-Go条件に該当しない
- Major懸念に対応方針が明確
- テスト計画が承認済み

## Best Practices

### 効果的な会議のために

1. **会議タイプを正しく選択**: Design or Implementation
2. **課題を明確に**: Phase 1で曖昧さを排除
3. **多様な視点を尊重**: 全エージェントの意見を聞く
4. **強制懸念を活用**: 見落としを防ぐ最重要ルール
5. **投票で決着**: 議論が平行線の場合は投票へ
6. **イテレーションを恐れない**: 2回までは許容
7. **外部レビューを活用**: 異なる視点を取り入れる
8. **🆕 テスト計画を先に**: 実装前にテスト戦略を確定

### アンチパターン

| パターン | 問題 | 対策 |
|---------|------|------|
| 全員一致症候群 | 見落としが増える | 強制懸念ルール |
| 無限イテレーション | 進捗しない | 最大2回制限 |
| 投票スキップ | 決定が曖昧 | Phase 8必須 |
| 外部レビュー無視 | 盲点が残る | 統合プロセス |
| 🆕 テストなし実装 | 品質低下 | AIP Phase 4必須 |
