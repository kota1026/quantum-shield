# Agent Meeting Protocol v3.2

> **Document Version**: 3.2  
> **Last Updated**: 2025-12-21  
> **Purpose**: 11エージェントによる設計会議の標準プロセス

---

## Overview

Agent Meeting Protocol は、11人のAIエージェントが協力して複雑な設計課題を解決するための構造化されたプロセスである。各エージェントは専門的な視点を持ち、提案、批評、投票を通じて最適な設計に到達する。

### Key Features

- **強制懸念提出**: 各エージェントが最低1つの懸念を提出必須
- **機能別投票**: 全機能を個別に投票で確定
- **イテレーション**: 重大懸念は追加Roundで解決
- **外部レビュー統合**: ChatGPT/Gemini等の外部AIの批評を統合可能

## Phase Details

### PHASE 1: 課題提示

**目的**: 解決すべき課題を全エージェントに明確に伝える

**フォーマット**:
```
┌─────────────────────────────────────────────────────────────────┐
│  課題: [課題タイトル]                                           │
│                                                                 │
│  現状の問題:                                                    │
│  ├── [問題1]                                                   │
│  ├── [問題2]                                                   │
│  └── [問題3]                                                   │
│                                                                 │
│  目標:                                                          │
│  ├── [目標1]                                                   │
│  ├── [目標2]                                                   │
│  └── [目標3]                                                   │
│                                                                 │
│  制約:                                                          │
│  ├── [制約1]                                                   │
│  └── [制約2]                                                   │
│                                                                 │
│  参考:                                                          │
│  └── [既存設計、競合、学術論文等]                              │
└─────────────────────────────────────────────────────────────────┘
```

### PHASE 2: 提案競争

**目的**: 多様な視点からの解決策を集める

**ルール**:
| ルール | 説明 | ポイント |
|--------|------|---------|
| 独自提案必須 | 他エージェントへの「同意」のみは禁止 | -10pt |
| 具体性重視 | 抽象的提案より具体的提案を評価 | - |
| 提案採用 | 最終設計に採用された場合 | +20pt |
| 部分採用 | 一部が採用された場合 | +10pt |
| 新規アイデア | 議論に新しい視点を提供 | +8pt |

**フォーマット**:
```
### [Agent名]
> **提案**: [提案タイトル]

[提案内容]

理由:
├── [理由1]
└── [理由2]
```

### PHASE 3: 相互批評

**目的**: 提案の弱点を発見し、改善する

**ルール**:
| ルール | 説明 | ポイント |
|--------|------|---------|
| Red Team必須反論 | Red Teamは全提案に反論必須 | 反論なし: -15pt |
| 建設的批判 | 批判は代替案を含むこと推奨 | +5pt |
| 重大欠陥発見 | 致命的な問題を発見 | +15pt |

### PHASE 4: 統合・投票

**目的**: 競合する選択肢を投票で解決

**投票ルール**:
- 全11エージェントが投票
- 過半数（6票以上）で決定
- 同数の場合: Red Team の投票を優先
- 棄権: 不可（必ず選択）

### PHASE 5: シーケンス作成

**目的**: 投票結果を反映したシーケンス図を作成

**形式**: ASCII art シーケンス図

**例**:
```
User            L3 Aegis            L1 Vault
  │                 │                   │
  │──(1) Request───►│                   │
  │                 │──(2) Submit──────►│
  │                 │                   │
  │◄──(3) Result────│                   │
```

### PHASE 6: 全体レビュー

**チェックリスト**:

| カテゴリ | チェック項目 |
|---------|-------------|
| 経済性 | Gas代は現実的か？手数料は競争力があるか？ |
| 攻撃ベクトル | 新たな攻撃が可能になっていないか？ |
| UX | 待機時間は許容範囲か？操作は複雑すぎないか？ |
| 依存性 | 外部システム（VRF等）障害時の影響は？ |
| 理念整合 | Core Principlesと矛盾していないか？ |
| 法的 | 規制上の問題はないか？ |
| 実装 | 技術的に実装可能か？ |

### PHASE 7: 課題再発見（強制版）

**目的**: 見落とされた問題を発見する

**強制ルール**:
```
┌─────────────────────────────────────────────────────────────────┐
│  ⭐ 強制懸念提出ルール                                          │
│                                                                 │
│  1. 各エージェントは最低1つの懸念点を提出しなければならない    │
│  2. 「問題なし」「特になし」は禁止（-10pt）                     │
│  3. 的外れな懸念でも減点しない（萎縮防止）                      │
│  4. 懸念点が後で実際の問題になった場合: +15pt                   │
│                                                                 │
│  目的: 集団思考を防ぎ、見落としを減らす                        │
└─────────────────────────────────────────────────────────────────┘
```

**分類基準**:

| レベル | 記号 | 定義 | アクション |
|--------|------|------|-----------|
| 致命的 | 🔴 | プロトコル存続に関わる | 次Round必須 |
| 重大 | 🟠 | 重要だが対策可能 | Round X.5で対策 |
| 中 | 🟡 | リスクはあるが許容範囲 | リスク認識して続行 |
| 軽微 | ⚪ | 影響が小さい | 認識のみ |

**フォーマット**:
```
| # | エージェント | 懸念 | 深刻度 | 対策 |
|---|-------------|------|--------|------|
| 1 | Cryptographer | [懸念内容] | 🟠 | [対策案] |
| 2 | CTO | [懸念内容] | 🟡 | 認識 |
...
```

**判定フロー**:
```
懸念点を集計
    │
    ├── 🔴 1件以上 → 次Round（設計やり直し）
    │
    ├── 🟠 1件以上 → Round X.5（対策追加してから続行）
    │       │
    │       └── 対策を追加 → Phase 8へ
    │
    └── 🟡 のみ → Phase 8へ進む
```

**イテレーション制限**:
- 最大2回のイテレーション（Round N → N.5 → 次課題）
- 無限ループ防止のため、2回目で🟠以下ならPhase 8へ強制進行

### PHASE 8: 機能別最終投票

**目的**: 全ての設計決定を民主的に確定

**対象**: 
- 全ての設計パラメータ
- 競合する選択肢
- 新規に議論された機能

**投票テンプレート**:
```
### 🗳️ 投票 #N: [機能名]

機能: [機能の説明]
選択肢:
  A. [選択肢A]
  B. [選択肢B]
  C. [選択肢C]

投票結果:
├── A: X票 (Agent1, Agent2, Agent3, ...)
├── B: Y票 (Agent4, Agent5, ...)
└── C: Z票 (Agent6, ...)

決定: [A/B/C]. [採用された選択肢の説明]
```

### PHASE 9: 最終シーケンス作成

**目的**: 投票結果を反映した完全なシーケンスを作成

**要件**:
- Phase 8で決定された内容のみ使用
- 未決定の機能は含めない
- 矛盾チェック必須
- 全フローを網羅

**出力物**:
- 主要フローのシーケンス図
- 各ステップの説明
- エラーハンドリング
- タイムライン

## Best Practices

### 効果的な会議のために

1. **課題を明確に**: Phase 1で曖昧さを排除
2. **多様な視点を尊重**: 全エージェントの意見を聞く
3. **強制懸念を活用**: 見落としを防ぐ最重要ルール
4. **投票で決着**: 議論が平行線の場合は投票へ
5. **イテレーションを恐れない**: 2回までは許容
6. **外部レビューを活用**: 異なる視点を取り入れる

### アンチパターン

| パターン | 問題 | 対策 |
|---------|------|------|
| 全員一致症候群 | 見落としが増える | 強制懸念ルール |
| 無限イテレーション | 進捗しない | 最大2回制限 |
| 投票スキップ | 決定が曖昧 | Phase 8必須 |
