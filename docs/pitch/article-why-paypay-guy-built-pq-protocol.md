# 決済会社の人間が、AIで「耐量子暗号プロトコル」を作った話

## はじめに — 俺は技術者じゃない

先に言っておく。俺はエンジニアではない。

本業はPayPayでBinanceとの暗号資産サービスの提携を推進している、いわゆる「ビジネスサイド」の人間だ。以前は日銀のCBDC（中央銀行デジタル通貨）フォーラムにPayPayの代表として参加していた。Web3領域ではUPBONDやアーリーワークスでも経験を積んだ。

つまり、金融と暗号資産の「交差点」にずっと立っている。でもコードを書くのは本業ではない。

そんな俺が、2025年11月に「量子コンピュータがブロックチェーンを壊す日」について考え始めて、4ヶ月後には34万行のコードベース、Ethereum Sepoliaにデプロイ済みのスマートコントラクト、155個の統合テストが全パスするプロトコルを完成させた。

AIと一緒に。

---

## 「あれ、これヤバくない？」— 2025年11月

きっかけは業務の延長だった。

PayPayでBinanceとの提携を進める中で、暗号資産の「保管」について深く考える機会が増えた。日銀CBDCフォーラムでも、デジタル通貨のセキュリティ基盤が議論のテーマに上がっていた。

ある日、NISTが2024年8月に耐量子暗号の最終標準を発表したことを知った。FIPS 204（Dilithium）、FIPS 205（SPHINCS+）。それ自体は技術ニュースとして流れてきただけだったが、逆に思った。

**「標準が出たということは、既存の暗号が本当に壊れる前提で世界が動き始めたということだ」**

そしてEthereumを見た。$2.5T以上のTVLが、すべてECDSA署名で守られている。量子コンピュータがECDSAを破れるようになったら、すべてのウォレットの秘密鍵が公開鍵から逆算できる。

Ethereumの耐量子化はVitalikの分析で2028年予定。でも、**今日ロックされている資産は、今日の暗号で守られている**。移行が完了するまでの2-3年間、何が資産を守るのか？

その隙間を埋めるプロトコルが必要だと思った。

## 設計 — 11体のAIエージェントに「会議」をさせた

ここが一番驚かれる部分だと思う。

プロトコルの設計を、**俺が1人で書いたわけではない**。かと言って、人間のチームがいたわけでもない。

やったのは、**11体のAIエージェントによる戦略会議**だ。

CTO、CSO（セキュリティ責任者）、CFO、Chief Cryptographer、Red Team...。それぞれ異なる視点を持つ11体のAIエージェントに役割を割り当て、9フェーズの構造化された会議プロセスで議論させた。

```
PHASE 1: 課題提示（5分）
PHASE 2: 提案競争（15分）← 全員提案必須。沈黙は-5pt
PHASE 3: 相互批評（10分）← Red Teamは全提案に批判必須
PHASE 4: 統合・投票（10分）← 賛成 > 反対+2 で可決
PHASE 5: シーケンス作成（15分）
PHASE 6: 全体レビュー（10分）
PHASE 7: 課題再発見（10分）← 全員懸念提出必須。提出なし=-10pt
PHASE 8: 機能別最終投票（10分）
PHASE 9: 最終シーケンス提示（5分）
```

このプロセスを**約10回**回した。

例えば「ユーザーの資産をアンロックする時、どういうプロセスにすべきか？」という問いに対して：

- **CTO**が「24時間のタイムロックを挟む」と提案
- **CSO**が「Proverが応答しない場合の緊急パスが必要」と指摘
- **CFO**が「緊急アンロックに保証金を要求してスパムを防ぐべき」と追加
- **Chief Cryptographer**が「SPHINCS+のハッシュベース署名なら格子暗号が破られても安全」と提案
- **Red Team**が「Prover 3体が結託したらどうする？」と攻撃
- **CTO**が「二次関数スラッシング（N²×10%）で結託コストを指数的に上げる」と応答

こうして、9つのコアシーケンスが固まった：

1. **ロック** — ユーザーが資産をVaultに預ける
2. **通常アンロック** — 24時間の待機期間 + VRFで選ばれたProverによる署名検証
3. **緊急アンロック** — Proverが反応しない場合、7日間 + 5%保証金
4. **Prover登録** — ステーキングで参加する検証者
5. **チャレンジ** — 不正を見つけたら報告（報酬あり）
6. **スラッシング** — 不正をしたProverは二次関数的にペナルティ（N²×10%）
7. **ガバナンス** — veQSトークン投票で運営
8. **緊急停止** — 72時間の一時停止機能
9. **トークンハブ** — veQSトークンの管理と報酬分配

なぜ2つの耐量子アルゴリズムを使うのか。これもエージェント会議で決まった。**Chief Cryptographerが「格子暗号（Dilithium）とハッシュ暗号（SPHINCS+）は異なる数学的前提に基づいている。片方が破られても、もう片方で守れる」と提案し、Red Teamが反論できなかった**。NISTが高価値システムに推奨している「defense-in-depth」そのもの。

俺の役割は何だったか。**会議の議題を設定し、インプットとなるドメイン知識（決済インフラのリスク、CBDCの要件、規制環境）を与え、最終的な採択判断をすること**。人間の経営者がやることと同じだ。ただし、テーブルの向こう側にいるのが11体のAIだっただけ。

## Claude Codeとの出会い — 「これ、作れるかも」

設計を書いたものの、Rust、Solidity、Next.js、PostgreSQL... 実装に必要な技術スタックは膨大だった。

そこでClaude Code（Anthropicが提供するCLIエージェント）を使い始めた。最初は半信半疑だったが、設計ドキュメントを読ませてコードを生成させると、思っていたよりはるかにまともなものが出てきた。

ただし、「AIに丸投げ」ではまったく動かない。

### 俺がやったこと（人間の仕事）

- **「何を作るか」の意思決定**: 量子脅威の隙間を見つけ、プロトコルを構想
- **AIエージェント会議の設計**: 11体のエージェント、9フェーズの会議プロトコル、インセンティブ/ペナルティ設計
- **会議の議題設定とドメイン知識注入**: 「決済インフラではこういうリスクがある」「CBDCではこういう要件がある」

- **統合判断と最終意思決定**: 何を先に作るか、どの提案を採択するか、エージェントが割れた時の裁定

### AIがやったこと

- **プロトコル設計**: 11体エージェントが10回の戦略会議で9シーケンスを定義
- **コード生成**: Rust/Axumバックエンド、Next.jsフロントエンド、Solidityコントラクト

- **リファクタリング**: 1,280箇所のモックデータ除去

## AIエージェント開発で学んだこと

### 1. AIは「会議させる」と良い設計が出る

AIに「耐量子暗号プロトコルを設計して」と言っても、教科書的な答えしか出てこない。しかし、11体に異なる役割を与えて**議論させる**と、1体では出てこないアイデアが生まれる。

Red Teamが「Prover 3体が結託したら？」と攻撃し、CTOが「二次関数スラッシングで結託コストを上げる」と応答する。この掛け合いは人間のチームミーティングと本質的に同じだ。

**要件定義すら、人間が全部やる必要はない。ただし「問い」を立てるのは人間の仕事。** これが4ヶ月で身につけた最大の教訓。

### 2. 設計と実装で「AIチーム」を分ける

設計フェーズは11体の戦略エージェント。実装フェーズでは、別の44個の専用プロンプトで「開発チーム」を構成した。

```
設計チーム（11体の戦略会議）:
  CTO / CSO / CFO / Chief Cryptographer / Red Team ...

開発チーム（44個の実装プロンプト）:
  00_master_orchestrator  — 全体統括
  03_impl                 — 実装
  04_review               — コードレビュー
  30_ui_impl              — UI実装
  32_i18n_audit           — 国際化チェック
  33_a11y_check           — アクセシビリティ検証
  37_e2e_test             — E2Eテスト
  70_layer_integration    — レイヤー統合
  80_fix_execution        — バグ修正
```

人間の組織と同じで、**設計チームと開発チームは分けた方がいい**。全部を1つのAIにやらせるとコンテキストが混乱する。

### 3. ProverもAIにした — 「人間がいないプロトコル」

ここまで来ると、もう一つ思いついた。

Quantum Shieldのプロトコルには「Prover」という役割がいる。ユーザーがロックした資産をアンロックする時、Proverが署名検証して承認する第三者だ。当初の設計では人間のオペレーターがProverを運営する想定だった。

**でも、Proverも AIにすればいいのでは？**

そう思って作ったのが `AI Prover Agent` だ。

```
Unlockリクエストが来る
  ↓
AIが内容を分析（金額、タイミング、署名検証、パターン）
  ↓
信頼度 >= 99%  → AUTO_SIGN（自動署名）
信頼度 >= 80%  → ESCALATE（人間に通知）
信頼度 <  80%  → REJECT（拒否 + セキュリティアラート）
```

中身はClaude APIを呼んで、Unlockリクエストの「これは署名していいか？」を判断させている。100 ETHを超える大口、期限切れ間近、Dilithium署名未検証などの異常パターンを検出し、人間に判断を仰ぐかAIが自動で処理するかを決める。

つまり：
- **設計**: AIエージェント11体が会議で決めた
- **実装**: AIが書いた
- **運用**: AIが署名検証する

**プロトコルの設計、実装、運用の全レイヤーにAIがいる。** 人間（俺）がやったのは「何を作るか」を決めて、AIの「組織」を設計して、最終的な意思決定をすることだけだ。

### 4. 「動くモック」と「本物」の境界線

最大の苦労は統合だった。AIは個別のコンポーネントは上手く作れるが、全体を繋げるのは人間がやるしかない。フロントエンドとバックエンドを繋ぎ、データベースに実際にデータが入り、L1で実際にトランザクションが通ることを確認する作業。

1,280箇所のモックデータ（FALLBACK_*, MOCK_*）を全て本物のAPI呼び出しに置き換えるのに、約1ヶ月かかった。

## そして地殻変動が起きた

2026年1月24日。俺がフロントエンドとバックエンドの統合に苦戦している最中に、Ethereum Foundationが動いた。

**「Post-Quantum SecurityをEFのトップ戦略優先事項とする」**

Justin Drakeが宣言し、Thomas Coratger率いるPQ専任チームが発足。$2Mの研究賞金。隔週のAll Core Developers PQ breakoutコール開始。

そして2月、ZKnoxが設立された。Ledger元CTOのNicolas Bacca、暗号学者2名のチーム。EFの公式バック付き。NTT（数論変換）をYulで最適化し、FALCON署名検証のガスコストを24M→2Mに削減。EIP-8051、EIP-8052を提出。

俺が1人でAIと格闘してる間に、元Ledger CTOが3人チームでEFの資金付きで同じ領域に入ってきた。

正直に言う。「あ、終わったかも」と思った。

## 終わったのか？

冷静に比較すると、実はやっていることが違う。

| | ZKnox | 俺 |
|--|-------|-----|
| アプローチ | NTTをYulで最適化 → on-chainで直接検証 | ZK証明でoff-chain検証 → on-chainはproofだけ |
| ガスコスト | 2M gas（12x改善） | ~数百K gas（proof検証のみ） |
| on-chainデータ | フル署名（2.4KB） | 260バイト |
| NIST標準 | 独自変種（ETHFALCON）に改変 | **FIPS 204そのまま** |
| 汎用性 | FALCON/Dilithium限定 | 任意のPQアルゴリズムに対応可能 |
| チーム | 3人 + EF資金 | 1人 + Claude Code |
| 認知度 | EF公式バック | ゼロ |

技術的には補完関係にある。でも認知度の差は圧倒的だ。

## では、何が残るのか

4ヶ月の開発で、少なくとも以下は確実に言える：

**1. 非エンジニアでも、AIを使えば暗号プロトコルを作れる時代になった**

これは俺が証明した。34万行、155テスト全パス、テストネットデプロイ済み。「作れるか」という問いに対する答えはYESだ。

**2. 「作れる」と「勝てる」は全く別の話**

Ledger元CTOと3人の暗号学者がEFの資金付きで参入してきた時点で、同じ土俵で戦うのは非合理的だ。これもビジネスパーソンならわかる判断。

**3. 本当の価値は「交差点」にいること**

決済会社にいて、CBDCフォーラムに出ていて、Binance提携をやっていて、しかも耐量子暗号プロトコルのプロトタイプを自分で（AIと）作れる人間。この交差点にいる人は、たぶん世界に俺しかいない。

技術で勝負するのではなく、**金融インフラの耐量子化を語れるビジネスパーソン**として動く方が、はるかに希少価値が高い。

## おわりに — AI時代の「作ってみた」の価値

「決済会社の人間が、AIで耐量子暗号プロトコルを作った」

この一文が、いま一番面白いと思っている。

技術者がAIを使って効率化するのは当たり前の話だ。でも、非技術者がAIを使って「本来作れないはずのもの」を作ることの意味は、もっと大きい。

だから、全部公開することにした。

**Quantum Shieldのコード、設計ドキュメント、AIプロンプト、全て。**

34万行のRust + Solidity + Next.js。11体のAIエージェントの設計プロンプト。44個の実装プロンプト。AI Prover Agentのコード。CLAUDE.md。全部だ。

なぜか？

「AIで作りました」と言うだけなら誰でもできる。「全部見せます」が一番の証明だからだ。コードの品質も、設計の意図も、失敗も、モックだらけだった過去のコミット履歴も、全部見てもらえばいい。

それに、隠す理由がない。技術で勝負するつもりなら隠す意味があるが、俺の価値は「交差点にいること」だ。コードは公開した方が、その交差点の希少性がより伝わる。

もし耐量子暗号に興味がある人、AIでプロトコルを作ろうとしている人、あるいは金融インフラの未来を考えている人がいたら、Star投げてくれると励みになる。IssueもPRも歓迎。
